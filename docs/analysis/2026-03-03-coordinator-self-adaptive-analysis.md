## 一、现状总结

### 已有能力

| 版本 | 能力 | 说明 |
|------|------|------|
| 2.47.0 | 任务理解自动化 | 自动识别任务类型、复杂度、影响范围、置信度 |
| 2.48.0 | 执行监控增强 | 进度预警、偏差检测、质量前移 |

### 当前局限

1. **流程选择是静态的**：基于预设规则（if-else），不是动态决策
2. **经验没有积累**：每次任务独立执行，历史经验只在"事后分析"中使用
3. **没有实时学习**：coordinator-optimization 是事后优化，不是实时决策

## 二、方向一：自适应流程

### 2.1 问题分析

#### 表面问题
- 当前流程选择基于任务类型的静态配置
- 跳过规则是预设的，不能根据实际情况调整

#### 根本原因
- 任务理解自动化只输出了"画像"，没有连接"流程决策"
- 流程选择逻辑是硬编码的，不是基于数据驱动的
- 缺乏"执行过程中的动态调整"能力

#### 关联因素
- 任务特征：类型、复杂度、影响范围
- 用户意图：quietMode、unattendedMode
- 执行反馈：进度、产出、质量
- 历史经验：类似任务的处理方式

### 2.2 核心思路

#### 2.2.1 什么情况下简化流程？

| 场景 | 简化方式 | 示例 |
|------|---------|------|
| **重复任务** | 使用历史流程，跳过探索 | 之前做过的功能，再次添加类似字段 |
| **简单任务** | 减少验证环节 | 修改配置、文档更新 |
| **紧急任务** | 跳过非必要环节 | 用户说"赶紧跑通" |
| **低风险** | 减少审查环节 | 实验性代码、内部工具 |

**简化判断逻辑**：
```
简化条件 = 满足以下任一条件：
- 历史相似度 >= 0.9（之前做过几乎一样的任务）
- 任务复杂度 <= 2 且影响范围 = single-file
- 用户明确要求简化（紧急/快速）
- 代码状态标记为"实验/临时"
```

#### 2.2.2 什么情况下加强流程？

| 场景 | 加强方式 | 示例 |
|------|---------|------|
| **新领域** | 增加调研、评审环节 | 首次接触的技术栈 |
| **高风险** | 增加审查、安全检查 | 涉及认证、支付、数据 |
| **复杂任务** | 增加里程碑、验证点 | 大型功能重构 |
| **不确定性** | 增加探索、确认环节 | 需求不明确 |

**加强判断逻辑**：
```
加强条件 = 满足以下任一条件：
- 任务复杂度 >= 4
- 影响范围 = system
- 涉及敏感操作（认证/支付/权限）
- 历史上此类任务失败率 > 30%
- 执行过程中发现意外复杂度
```

#### 2.2.3 如何判断任务大小/紧急程度？

**任务大小评估**（多维度加权）：

| 维度 | 指标 | 权重 |
|------|------|------|
| 代码量 | 涉及文件数、改动行数 | 30% |
| 依赖复杂度 | 模块依赖数、接口依赖数 | 25% |
| 业务复杂度 | 业务规则数、状态数 | 25% |
| 历史相似度 | 与历史任务的相似度 | 20% |

**紧急程度判断**：
```
紧急 = 用户明确说明（"赶紧"、"立即"）
    OR 执行时间窗口（"下班前"）
    OR 阻塞其他任务
```

#### 2.2.4 流程动态调整的触发条件

| 触发时机 | 检测内容 | 调整动作 |
|---------|---------|---------|
| 任务开始 | 任务特征 → 选择初始流程 | 设定基础流程 |
| 执行中检查点 | 进度偏差、产出偏差 | 调整后续环节 |
| 里程碑完成 | 质量评估、风险评估 | 加强/简化后续 |
| 用户反馈 | 新的指示 | 重新评估流程 |

### 2.3 具体实现方案

#### 2.3.1 流程决策引擎（伪代码）

```python

            optimized_flow = self._apply_history_optimization(
                base_flow,
                history_experience

        else:
            optimized_flow = base_flow

        # 4. 考虑用户意图
        final_flow = self._apply_user_intent(optimized_flow, user_context)
```

#### 2.3.2 动态调整机制

```
【动态调整流程】

任务开始
    ↓
[特征提取] → 任务画像
    ↓
[历史匹配] → 相似经验
    ↓
[流程决策] → 初始流程配置
    ↓
执行循环（里程碑制）
    ├─ 里程碑N完成
    │   ├─ [进度检查] → 正常/滞后
    │   ├─ [产出检查] → 符合/偏离
    │   ├─ [质量检查] → 达标/风险
    │   └─ [决策调整] → 是否调整后续流程
    │
    └─ 里程碑N+1
        ↓
任务完成 → 记录执行数据 → 供学习系统使用
```

#### 2.3.3 与现有系统的集成点

| 现有模块 | 集成点 | 集成方式 |
|---------|--------|---------|
| 任务理解自动化 | 输入 | 使用任务画像作为决策输入 |
| 执行监控增强 | 反馈 | 使用监控数据触发流程调整 |
| 跳过规则 | 替代 | 动态流程决策替代静态跳过规则 |
| quietMode/unattendedMode | 继承 | 用户意图作为决策参数 |

### 2.4 预期收益

| 收益 | 说明 |
|------|------|
| **效率提升** | 简单任务流程缩短 30-50% |
| **质量保障** | 复杂任务加强关键环节 |
| **灵活性** | 适应不同场景，不再僵化 |
| **用户满意度** | 紧急任务快速响应 |

### 2.5 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 简化导致质量下降 | 设置质量底线（不可跳过的检查点） |
| 调整逻辑过于复杂 | 渐进式引入，先从简单场景开始 |
| 决策不准确 | 引入置信度，低置信度时保持原流程 |

## 三、方向二：学习进化机制

### 3.1 问题分析

#### 表面问题
- 经验没有积累，每次都从零开始
- coordinator-optimization 只是事后分析

#### 根本原因
- 没有结构化的经验存储
- 没有实时学习机制（只有事后优化）
- 经验没有应用到决策中

#### 关联因素
- 执行记录：每次任务的输入、输出、决策
- 成功/失败模式：什么情况下成功/失败
- 用户反馈：用户对结果的满意程度

### 3.2 核心思路

#### 3.2.1 应该学习什么？

**1. 成功经验（What worked）**
- 哪些流程组合成功了
- 哪些 Subagent 组合效率高
- 什么样的任务画像对应什么样的流程

**2. 失败教训（What failed）**
- 哪些流程组合失败了
- 失败的原因是什么
- 什么预警信号预示失败

**3. 决策上下文（Context）**
- 任务特征（类型、复杂度、范围）
- 用户意图（quietMode、unattendedMode）
- 执行环境（时间、状态）

#### 3.2.2 如何存储和检索？

**存储结构**：

```python
# 经验记录结构
class ExecutionExperience:
    task_profile: TaskProfile      # 任务画像
    flow_config: FlowConfig         # 使用的流程配置
    execution_result: Result        # 执行结果
    metrics: ExecutionMetrics       # 指标（时间、质量、轮次）
    lessons: List[Lesson]           # 经验教训
    timestamp: datetime             # 时间戳
```

**检索方式**：

| 检索方式 | 适用场景 | 实现方式 |
|---------|---------|---------|
| 精确匹配 | 完全相同的任务 | 任务画像哈希 |
| 相似匹配 | 类似任务 | 向量相似度 |
| 模式匹配 | 已知模式 | 规则匹配 |

#### 3.2.3 学习结果如何应用到决策？

```
【应用流程】

新任务进入
    ↓
提取任务画像 → 向量化
    ↓
检索相似经验（Top-K）
    ↓
评估相似经验
    ├─ 高相似度（>=0.9）→ 直接应用经验
    ├─ 中相似度（0.7-0.9）→ 部分应用 + 标记需观察
    └─ 低相似度（<0.7）→ 探索模式，记录新经验
    ↓
决策时参考经验
    ↓
执行 → 记录结果 → 更新经验库
```

**置信度机制**：

| 相似度 | 应用策略 |
|--------|---------|
| >= 0.9 | 高置信度，直接应用 |
| 0.7-0.9 | 中置信度，建议性应用 |
| 0.5-0.7 | 低置信度，供参考 |
| < 0.5 | 无相似经验，标记为探索 |

#### 3.2.4 如何验证学习效果？

**1. 短期验证（单次任务）**
- 执行前：记录预期
- 执行后：对比预期 vs 实际

**2. 中期验证（5-10次任务）**
- 时间预估准确率
- 流程选择准确率
- 用户满意度

**3. 长期验证（系统性）**
- 相似任务成功率变化
- 决策质量趋势
- 优化建议采纳率

### 3.3 具体实现方案

#### 3.3.1 经验存储系统

```python
# 经验存储系统
class ExperienceStore:
    def __init__(self, storage_path):
        self.storage = JsonStorage(storage_path)

    def save(self, experience: ExecutionExperience):
        """保存经验"""
        # 1. 提取任务特征向量
        vector = self._extract_feature_vector(experience.task_profile)

        # 2. 保存到存储
        self.storage.append({
            'vector': vector,
            'experience': experience,
            'embedding': self._compute_embedding(vector)
        })

    def retrieve_similar(self, task_profile: TaskProfile, top_k=5):
        """检索相似经验"""
        # 1. 提取任务特征向量
        vector = self._extract_feature_vector(task_profile)

        # 2. 计算与存储中所有经验的相似度
        similarities = []
        for item in self.storage:
            sim = cosine_similarity(vector, item.embedding)
            similarities.append((sim, item))

        # 3. 返回 Top-K
        return sorted(similarities, key=lambda x: x[0], reverse=True)[:top_k]
```

#### 3.3.2 学习更新机制

```python
# 学习更新机制
class LearningSystem:
    def __init__(self, experience_store):
        self.experience = experience_store
        self.patterns = PatternExtractor()

    def on_task_complete(self, task_execution: TaskExecution):
        """任务完成后学习"""
        # 1. 提取经验
        experience = self._extract_experience(task_execution)

        # 2. 分析成功/失败模式
        if task_execution.success:
            self.patterns.add_success_pattern(experience)
        else:
            self.patterns.add_failure_pattern(experience)

        # 3. 保存经验
        self.experience.save(experience)

    def get_recommendation(self, task_profile: TaskProfile) -> Recommendation:
        """获取决策建议"""
        # 1. 检索相似经验
        similar = self.experience.retrieve_similar(task_profile)

        # 2. 分析模式
        patterns = self.patterns.query(task_profile.type)

        # 3. 生成建议
        return self._generate_recommendation(similar, patterns)
```

## 五、总结

### 推荐方案

**自适应流程 + 学习进化机制是互补的系统**：
- 自适应流程是"执行层面"的优化
- 学习进化是"学习层面"的优化
- 两者结合形成闭环：执行→学习→优化→更好执行

**建议实施优先级**：
1. 先实现经验存储（基础设施）
2. 再实现流程决策框架（基于规则）
3. 然后引入学习机制（数据驱动）
4. 最后实现动态调整（实时优化）

### 关键成功因素

1. **数据质量**：经验记录需要准确、完整
2. **渐进引入**：不要一次性引入太多复杂性
3. **可观测性**：决策过程可追溯、可解释
4. **用户反馈**：用户满意度是最终指标
