# 需求文档：角色体系架构设计

> 创建时间：2026-02-26

## 1. 背景

当前 AI-Assistant 的 Skills 是按"能力类型"划分，导致：
- 技能边界模糊
- 职责混杂
- 难以复用

需要重新设计为**按角色划分**的架构。

---

## 2. 目标

建立清晰的角色体系：
- 每个角色有明确职责
- 技能按最小职责拆分
- 管家协调角色执行任务

---

## 3. 角色体系

### 3.1 核心角色（线性流程）

| 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|
| 需求分析师 | 理解用户真正想要什么 | 用户想法 | 需求文档 |
| 计划制定者 | 把需求变成可执行计划 | 需求文档 | 实施计划 |
| 执行者 | 按计划实现代码 | 实施计划 | 实现代码 |
| 质量保证 | 验证实现是否符合需求 | 实现代码 | 验收报告 |
| 代码审查 | 确保代码质量 | 验收报告 | 完成总结 |

### 3.2 辅助角色（按需调用）

| 角色 | 职责 | 场景 |
|------|------|------|
| 调度员 | 判断用什么角色 | 任何任务开始时 |
| 思维教练 | 思维方式指导 | 遇到思维困惑时 |
| 调试专家 | Bug 定位修复 | 代码出错时 |
| 代码分析师 | 代码问题分析 | 需要代码分析时 |
| 安全审查 | 安全问题审查 | 有安全问题时 |
| 测试设计 | 测试用例设计 | 需要测试时 |

## 4. 技能原子化设计

### 4.1 技能拆分原则

- 每个 Skill 只负责一个最小职责
- 技能可被多个角色复用
- 边界清晰，不重复

## 6. 管家（Coordinator）职责

### 6.1 核心职责

**管家是"调度者"不是"执行者"**——只判断和分发，不亲自执行具体任务。

| 职责 | 说明 |
|------|------|
| 需求判断 | 分析用户真正想要什么 |
| 角色选择 | 判断需要哪些角色 |
| 方案制定 | 安排执行顺序和里程碑 |
| 任务分发 | 调用对应角色执行 |
| 进度监控 | 追踪执行状态 |
| 结果收集 | 汇总各角色产出 |
| 日志记录 | 记录执行过程 |

### 6.2 调整权限

| 调整类型 | 权限 | 需要用户确认？ |
|---------|------|--------------|
| 需求变更 | ❌ 不能处理 | ✅ 必须 |
| 里程碑变更 | ❌ 不能处理 | ✅ 必须 |
| 角色组合变更 | ❌ 不能处理 | ✅ 必须 |
| 任务顺序调整 | ✅ 可以 | ❌ 自动 |
| 任务细节修改 | ✅ 可以 | ❌ 自动 |

### 6.3 自我优化

**触发时机**：每完成一个里程碑

**优化层次**：
- 任务级别：分配是否合理、时间预估
- 阶段级别：角色组合效率、里程碑设置
- 系统级别：用户偏好、角色配置调整

**存储位置**：`docs/plans/coordinator-optimization.md`
