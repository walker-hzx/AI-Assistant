# v5.1 Agent 强化计划

> 核心思路：不再优化调度架构，而是逐个强化 Agent 和 Skill，弥补开发过程中的实际短板。

---

## 一、现状诊断

### 当前能力地图

```
开发者日常操作          对应能力              当前状态
─────────────────────────────────────────────────
思考/分析             analyst + /thinking     ✅ 完善
需求讨论             /discuss (内联)          ⚠️ 未加载 brainstorming skill
制定计划             /plan → writing-plans    ✅ 完善
执行计划             /executing-plans         ✅ 完善
写代码实现           executor                ❌ 无直接命令入口
测试                tester + /verification   ✅ 完善
代码审查             reviewer + /review       ✅ 完善
安全审查             /security-review         ✅ 完善
调试                debugger + /debugging    ✅ 完善
学习/调研            researcher              ⚠️ 命令太窄（仅 /learn-concept）
外部资源获取          scout + /docs-sync       ✅ 完善
重构                ???                     ❌ 完全缺失
写文档              ???                     ❌ 完全缺失
生成组件/端点脚手架   ???                     ❌ 完全缺失
```

### 发现的 6 类问题

#### 问题 A：命令覆盖不完整

executor 和 researcher 是日常开发最高频使用的两个 agent，但它们入口太窄或没有入口：

| Agent | 当前命令 | 问题 |
|-------|---------|------|
| executor | 无直接命令 | 只能通过 /executing-plans（要求先写计划）才能触达 |
| researcher | /learn-concept | 仅限"学习概念"，无法用于"分析这个模块"、"调研这个方案" |

**影响**：用户想让 executor 直接实现一个小功能，或让 researcher 分析一段代码，没有直达路径。

#### 问题 B：技能分配不精准

Agent 拥有的 skill 和它实际需要的 skill 之间有错位：

| Agent | 当前 Skills | 缺失 Skills | 影响 |
|-------|-----------|------------|------|
| executor | code-implementation, tdd | **vue3-vite-guide**, **python-fastapi-guide** | 写前端/后端代码时没有技术栈规范参考 |
| researcher | code-analysis, project-researcher | **web-researcher** | 需要查外部资料时无能力，只能分析本地代码 |
| analyst | 4 个需求分析 skill | **brainstorming** | 早期发散思考阶段缺少头脑风暴支持 |

#### 问题 C：缺失关键开发能力

三个高频开发场景完全没有对应能力：

| 缺失能力 | 使用场景 | 频率 |
|---------|---------|------|
| **重构** | 代码结构调整、函数提取、模块拆分、命名优化 | 高 |
| **文档编写** | API 文档、README、架构说明、注释 | 中 |
| **脚手架** | 新建组件、新建 API 端点、新建测试文件 | 中 |

#### 问题 D：孤立能力

brainstorming skill 存在但没有被任何 agent 或 command 引用，是一个"孤儿"能力。

#### 问题 E：/discuss 命令过于简陋

/discuss 是内联命令（不 fork agent，不加载 skill），只有一段简单提示词。作为需求讨论的入口，它应该更强——至少加载 brainstorming skill。

#### 问题 F：文档过时

- agents/README.md 仍引用 "Coordinator 根据任务复杂度选择 S/M/L 档"
- 根目录 CLAUDE.md 仍使用 coordinator 调度流程描述

## 二、优化方案

### Phase 1：补命令入口（低成本，立即见效）

> 目标：让每个核心 Agent 都有直达命令

| 动作 | 详情 |
|------|------|
| 新增 `/implement` 命令 | `context: fork, agent: executor` — 直接告诉 executor 实现一个功能 |
| 新增 `/analyze` 命令 | `context: fork, agent: researcher` — 让 researcher 分析代码/方案 |
| 改造 `/discuss` 命令 | 从内联改为 `skill: brainstorming`，加载头脑风暴能力 |

**工作量**：新建 2 个 md 文件 + 修改 1 个 md 文件

### Phase 2：补技能分配（低成本，能力提升明显）

> 目标：每个 Agent 有完整的技能武装

| Agent | 动作 | 原因 |
|-------|------|------|
| executor | 添加 `vue3-vite-guide` + `python-fastapi-guide` | 写代码时有技术栈规范参考 |
| researcher | 添加 `web-researcher` | 调研时可以搜索外部资料 |
| analyst | 添加 `brainstorming` | 早期发散思考有方法论支撑 |

**工作量**：修改 3 个 agent md 文件的 skills 列表

### Phase 3：补缺失能力（中等成本，填补关键盲区）

> 目标：覆盖重构、文档编写两个高频场景

#### 3a. 新增 refactoring skill

核心内容：
- 安全重构步骤（先写测试 → 小步重构 → 每步验证）
- 常见重构模式（提取函数/组件、内联、改名、移动、替换算法）
- 重构范围控制（不要一次改太多，逐步推进）
- 重构前检查清单（有测试吗？影响范围多大？）

归属：executor（重构本质上是写代码，由 executor 执行）

新增命令：`/refactor` → executor

#### 3b. 新增 doc-writing skill

核心内容：
- 文档类型识别（API 文档、README、架构说明、变更记录）
- 文档模板（不同类型有不同结构）
- 从代码生成文档的方法
- 文档质量标准（准确性、完整性、可读性）

归属：researcher（调研员对代码最熟悉，适合写文档）

新增命令：`/docs` → researcher

**工作量**：新建 2 个 skill 文件 + 2 个 command 文件 + 修改 2 个 agent 文件

### Phase 4：更新文档（收尾，保持一致性）

| 文件 | 动作 |
|------|------|
| agents/README.md | 重写，反映 v5.0 直达自治模式 |
| CLAUDE.md (根目录) | 更新开发流程，去掉 coordinator 引用 |
| .claude/CLAUDE.md | 更新命令路由表（添加新增命令） |

**工作量**：修改 3 个文档文件

## 三、实施顺序和优先级

```
Phase 1  ──→  Phase 2  ──→  Phase 3  ──→  Phase 4
补命令        补技能        补新能力        更新文档
~15min        ~10min        ~2h            ~30min
立即见效      能力连接      填补盲区        保持一致
```

### 为什么这个顺序？

1. **Phase 1 + 2 成本极低，效果立竿见影** — 只改配置，不写新内容
2. **Phase 3 是核心价值** — 真正填补开发中的能力短板
3. **Phase 4 是收尾** — 确保文档和实际一致

```
开发者日常操作          对应能力                    命令入口
─────────────────────────────────────────────────────────
思考/分析             analyst (+ brainstorming)    /thinking
需求讨论             brainstorming skill          /discuss
制定计划             writing-plans skill          /plan
执行计划             executing-plans skill        /executing-plans
写代码实现           executor (+ 技术栈指南)       /implement ← new
重构代码             executor (+ refactoring)     /refactor ← new
测试                tester                       /verification, /test-planner
代码审查             reviewer                     /review
安全审查             reviewer                     /security-review
调试                debugger                     /debugging
学习概念             researcher                   /learn-concept
```

**核心变化**：
- 命令覆盖率：12 → 16（补齐 executor/researcher 入口 + 重构 + 文档）
- 每个核心 Agent 都有至少 1 个直达命令
- 技能分配更精准（executor 有技术栈规范，researcher 有外部搜索）
- brainstorming 不再是孤儿能力

## 六、Phase 3 详细设计

### refactoring skill 内容大纲

```markdown
# 重构

> 安全、渐进地改善代码结构，不改变外部行为

## 重构前检查
- 有测试覆盖吗？没有先补测试
- 影响范围评估（几个文件？几个调用方？）
- 确认重构目标（为什么要重构？达到什么效果？）

## 重构流程
1. 确保所有测试通过（绿灯起步）
2. 单步重构（一次只做一种重构操作）
3. 每步后运行测试验证
4. 如果测试失败，立即回退这一步

## 常见重构操作
- 提取函数/方法
- 提取组件（前端）
- 内联临时变量
- 重命名（变量/函数/文件）
- 移动函数/类到更合适的模块
- 替换条件分支为策略模式
- 消除重复代码
- 简化嵌套（提前返回、guard clause）

## 重构范围控制
- 单次 PR 只处理一种重构
- 不在重构中混入功能改动
- 大范围重构分多次 PR
```

### doc-writing skill 内容大纲

```markdown
# 文档编写

> 从代码和上下文生成准确、有用的文档

## 文档类型
- README（项目/模块说明）
- API 文档（端点、参数、返回值）
- 架构文档（系统设计、模块关系）
- 变更记录（CHANGELOG）
- 使用指南（How-to）

## 编写流程
1. 确定文档类型和读者
2. 读取相关代码，提取核心信息
3. 按文档模板组织内容
4. 验证准确性（代码 vs 文档对照）

## 质量标准
- 准确性：与代码一致
- 完整性：覆盖核心功能
- 可读性：新人能看懂
- 简洁性：不写废话
```
