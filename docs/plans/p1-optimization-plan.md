## P1-1：明确"可选"步骤的触发条件

### 问题分析

当前 CHECKPOINT.md 标记以下为"可选"：
- 检查点 2：需求分析
- 检查点 3：制定计划
- 检查点 5：功能验证
- 检查点 6：代码审查

但没有明确什么情况下"可选"、什么情况下"必须"。

### 解决方案

增加"任务类型与检查点对照表"：

```markdown
## 【核心】任务类型与检查点对照

| 任务类型 | 必做检查点 | 可选检查点 |
|----------|------------|-------------|
| 调研任务 | 0, 1, 4, 8 | 2, 3 |
| Bug修复 | 0, 1, 2, 4, 5, 8 | 3, 6 |
| 功能开发 | 0, 1, 2, 3, 4, 5, 6, 8 | 7 |
| 复杂任务 | 0, 1, 2, 3, 4, 5, 6, 7, 8 | - |
| 重构任务 | 0, 1, 2, 3, 4, 5, 6, 8 | 7 |
| 安全审查 | 0, 1, 2, 4, 5, 6, 8 | 3, 7 |

### 触发条件说明

| 检查点 | 何时必做 | 何时可选 |
|--------|----------|----------|
| 需求分析 | 需求不清晰、复杂任务 | 需求明确、简单任务 |
| 制定计划 | 复杂任务、功能开发 | 简单任务、调研任务 |
| 功能验证 | 有功能产出的一切任务 | 纯调研任务 |
| 代码审查 | 功能开发、重构 | Bug修复、调研任务 |
| 质量评估 | 复杂任务 | 其他任务 |
```

## P1-2：增加内容质量检查

### 问题分析

当前只检查：
- 文档是否存在
- 必填字段是否存在

不检查：
- 内容是否充分
- 质量是否达标

### 解决方案

增加"内容质量检查清单"：

```markdown
## 【核心】内容质量检查

### 需求分析文档质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 功能目标 | 明确、具体、可测试 | 是否有明确的功能描述 |
| 业务规则 | 完整、无歧义 | 是否有遗漏的业务场景 |
| 验收标准 | 量化、可验证 | 是否有可测试的验收条件 |
| 风险识别 | 全面、有缓解措施 | 是否有风险点和应对方案 |

### 计划文档质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 任务拆分 | 2-5分钟粒度 | 每个任务是否可独立完成 |
| 依赖关系 | 清晰、无循环 | 任务间依赖是否正确 |
| 执行顺序 | 合理、最优 | 顺序是否符合逻辑 |
| 里程碑 | 明确、可交付 | 每个里程碑是否有产出 |

### 验证报告质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 测试覆盖 | 核心功能覆盖 | 是否有遗漏的关键功能 |
| 测试质量 | 可执行、有断言 | 测试是否真正验证功能 |
| 问题记录 | 完整、有截图 | 问题描述是否清晰 |

### 审查报告质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 问题识别 | 全面、准确 | 是否遗漏重要问题 |
| 严重程度 | 合理、正确 | 问题分级是否合理 |
| 修复建议 | 具体、可操作 | 建议是否可执行 |
```

## P1-3：统一 SKILL.md 与 CHECKPOINT.md 表述

### 问题分析

- SKILL.md 说"功能开发：0 → 1 → 2 → 3 → 4 → 5 → 6 → 8"（需要验证+审查）

### 解决方案

修改 CHECKPOINT.md，使其与 SKILL.md 一致：

```markdown
### 检查点 5：功能验证

**调研任务**：可选
**Bug修复**：必做
**功能开发**：必做
**复杂任务**：必做
**重构任务**：必做
**安全审查**：必做

### 检查点 6：代码审查

**调研任务**：不需要
**Bug修复**：可选
**功能开发**：必做
**复杂任务**：必做
**重构任务**：必做
**安全审查**：必做
```
