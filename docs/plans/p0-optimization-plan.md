# P0 问题优化方案

## 问题概述

| 优先级 | 问题 | 当前状态 | 目标状态 |
|--------|------|----------|----------|
| P0-1 | thinking-coach 深度思考能力不足 | 缺少系统性分析框架 | 具备完整的问题分析框架 |
| P0-2 | debugger 复杂场景处理能力薄弱 | 缺少场景化调试方法论 | 具备完整调试方法论 |

---

## P0-1：thinking-coach 优化方案

### 问题分析

**当前缺陷**：
1. 职责描述抽象，缺乏可操作的分析步骤
2. 没有系统性分析框架
3. 没有多维视角框架
4. 没有推理链条
5. 没有决策标准

**对比参考**：strategist 有完整的问题分析框架、深度分析维度、方案评估框架

### 优化方向

#### 1. 增加问题分析框架

参考 strategist 的问题分析框架，为 thinking-coach 增加：

```markdown
## 【核心】问题分析框架

### 分析步骤

1. **表面问题** - 用户描述的现象
2. **思维模式** - 用户当前的思维方式
3. **核心问题** - 思维层面的根本问题
4. **盲点识别** - 用户没看到的角度
5. **指导建议** - 具体的思维调整建议
```

#### 2. 增加多维视角框架

```markdown
## 【核心】多维视角框架

| 视角 | 问题 | 应用场景 |
|------|------|----------|
| 本质视角 | 真正的问题是什么？ | 所有场景 |
| 时间视角 | 短期 vs 长期 | 决策类 |
| 利益视角 | 各方的关切点 | 需求冲突 |
| 风险视角 | 可能的风险是什么？ | 方案评估 |
| 机会视角 | 问题背后的机会 | 困境类 |
| 系统视角 | 局部与整体的关系 | 复杂问题 |
```

#### 3. 强化决策点思考清单

当前只有简单的5步流程，需要增加类似 strategist 的决策点：

```markdown
### 决策点 1：接收问题后

**思考清单**：
- 用户的核心困惑是什么？
- 问题出在"不知道"还是"想错了"？
- 需要厘清思路还是纠正方向？

**思考方式**：
```
1. 用户是在问"怎么做"还是"是什么"？
2. 用户的思维卡在哪里？
3. 有什么背景信息我需要了解？
```

### 决策点 2：分析问题中

**思考清单**：
- 用户的思维方式有什么问题？
- 有什么盲点用户没看到？
- 正确的思考方向是什么？

### 决策点 3：给出建议前

**思考清单**：
- 建议是否针对核心问题？
- 用户能否理解和执行？
- 如何让用户自己找到答案？
```

#### 4. 增加思维方法论

```markdown
## 【核心】思维方法论

### 1. 5Why 分析法
连续追问"为什么"，找到根本原因

### 2. 假设验证法
提出假设，用证据验证

### 3. 视角转换法
从不同角度看问题

### 4. 逆向思维法
从结果倒推原因

### 5. 边界思维法
考虑极端情况和边界条件
```

## P0-2：debugger 优化方案

### 问题分析

**当前缺陷**：
1. 缺少具体调试技术方法（二分查找、日志分析、断点策略）
2. 缺少场景化调试路径（并发、异步、内存泄漏、性能问题）
3. 没有错误模式库
4. 边界覆盖不足

**对比参考**：test-designer 有完整的复杂场景方法论、场景模板、检查清单

### 优化方向

#### 1. 增加调试技术方法论

```markdown
## 【核心】调试技术方法论

### 方法 1：二分查找定位法
```
1. 找到问题发生的中点
2. 判断问题在左半边还是右半边
3. 重复直到定位到具体位置
```

### 方法 2：日志分析法
```
1. 收集相关日志
2. 找到第一个异常点
3. 追踪异常传播路径
4. 定位根因
```

### 方法 3：断点调试策略
```
1. 在问题发生前设置断点
2. 逐步执行，观察状态变化
3. 找到状态异常点
4. 分析原因
```

### 方法 4：最小复现法
```
1. 简化问题到最小可复现案例
2. 分析最小案例
3. 逐步增加复杂度
4. 找到触发条件
```
```

#### 2. 增加场景化调试路径

```markdown
## 【核心】场景化调试路径

### 场景 1：并发/竞态调试

**特征**：
- 随机出现
- 难以复现
- 状态不一致

**调试路径**：
1. 识别竞态条件
2. 添加日志记录时序
3. 使用锁或原子操作
4. 验证修复

**检查清单**：
- [ ] 是否有共享状态访问
- [ ] 是否有非原子操作
- [ ] 是否有锁粒度问题
- [ ] 是否有死锁风险

### 场景 2：异步/回调调试

**特征**：
- 回调未执行
- 回调顺序错误
- Promise 未 resolve

**调试路径**：
1. 检查回调是否被调用
2. 检查错误是否被捕获
3. 检查 Promise 状态
4. 验证异步流程

**检查清单**：
- [ ] 回调是否正确绑定 this
- [ ] 错误是否正确传播
- [ ] Promise 是否正确处理
- [ ] 异步时序是否正确

### 场景 3：内存泄漏调试

**特征**：
- 内存持续增长
- 性能逐渐下降
- GC 无法回收

**调试路径**：
1. 使用内存分析工具
2. 定位泄漏对象
3. 追踪对象引用链
4. 修复引用问题

**检查清单**：
- [ ] 是否有未清理的定时器
- [ ] 是否有未移除的事件监听
- [ ] 是否有闭包引用
- [ ] 是否有缓存无限增长

### 场景 4：性能问题调试

**特征**：
- 响应慢
- CPU 高
- 请求耗时

**调试路径**：
1. 使用性能分析工具
2. 定位热点代码
3. 分析算法复杂度
4. 优化性能瓶颈

**检查清单**：
- [ ] 是否有 N+1 查询
- [ ] 是否有不必要的重渲染
- [ ] 是否有大文件处理
- [ ] 是否有缓存未命中
```

#### 3. 增加错误模式库

```markdown
## 【核心】错误模式库

### 类型 1：空指针/未定义
**特征**：访问 null/undefined 的属性
**常见场景**：
- API 返回空值未处理
- 异步数据未加载
- 条件判断遗漏

### 类型 2：类型错误
**特征**：类型不匹配
**常见场景**：
- 隐式类型转换
- JSON 解析错误
- 数值运算错误

### 类型 3：异步错误
**特征**：异步操作失败
**常见场景**：
- Promise 未 catch
- 回调未处理错误
- async/await 未 try/catch

### 类型 4：逻辑错误
**特征**：代码执行但结果不对
**常见场景**：
- 条件判断错误
- 循环边界错误
- 状态管理错误
```

```markdown
- [ ] 回归测试

### 总结阶段
- [ ] 根因记录
- [ ] 经验总结
- [ ] 测试补充
- [ ] 文档更新
```
