# 质疑者机制细化方案

> 目标：在关键节点增加质疑环节，确保质量

---

## 一、质疑流程设计

### 1.1 质疑机制核心逻辑

```
【质疑流程】

产出完成后
    ↓
管家执行自检（对照检查清单）
    ↓
识别不确定项和需要确认的问题
    ↓
展示问题 → 询问用户确认
    ↓
用户确认"可以了" → 质疑通过
    ↓
继续下一步

【循环机制】
- 如果用户说"还有问题" → 继续追问
- 直到用户说"可以了" → 通过
- 最大循环次数：3次
```

## 二、各阶段质疑检查清单

### 2.1 步骤1后：需求质疑

**时机**：thinking-coach 分析完成后

**检查清单**：

| 序号 | 检查|------|--------|------|
| 项 | 问题 |
1 | 需求完整性 | 还有其他需求吗？ |
| 2 | 需求清晰度 | 这里有歧义吗？具体指什么？ |
| 3 | 边界条件 | 边界情况有哪些？ |
| 4 | 验收标准 | 验收标准清晰可测吗？ |
| 5 | 约束条件 | 有什么不要的/限制？ |

**质疑话术**：
```
【需求确认】

我理解您的需求是：<理解摘要>

为了确保理解正确，我有几个问题：

1. <问题1>
2. <问题2>
3. <问题3>

请您确认以上理解是否正确，或者补充其他需求。
```

---

### 2.2 步骤2后：需求分析质疑

**时机**：requirements-analyst 产出需求文档后

**检查清单**：

| 序号 | 检查项 | 问题 |
|------|--------|------|
| 1 | 功能完整 | 所有功能点都覆盖了吗？ |
| 2 | 边界覆盖 | 边界场景都有吗？ |
| 3 | 验收标准 | 每条标准可测试吗？ |
| 4 | 业务规则 | 业务规则都清晰吗？ |
| 5 | 风险识别 | 风险点都识别了吗？ |

**质疑话术**：
```
【需求分析确认】

需求文档已产出，我检查后发现：

1. 已覆盖：<列表>
2. 可能遗漏：<列表>
3. 需要澄清：<列表>

请您确认需求是否完整，或者补充其他内容。
```

---

### 2.3 步骤3后：计划质疑

**时机**：executor 产出计划文档后

**检查清单**：

| 序号 | 检查项 | 问题 |
|------|--------|------|
| 1 | 任务完整 | 所有需求都有对应任务吗？ |
| 2 | 任务粒度 | 每个任务2-5分钟可完成吗？ |
| 3 | 依赖关系 | 依赖顺序对吗？ |
| 4 | 里程碑 | 里程碑划分合理吗？ |
| 5 | 时间预估 | 时间预估合理吗？ |

**质疑话术**：
```
【计划确认】

计划已产出，我检查后发现：

1. 任务覆盖：<列表>
2. 可能遗漏：<列表>
3. 需要调整：<列表>

请您确认计划是否合理，或者需要调整。
```

---

### 2.4 步骤4后：产出质疑

**时机**：executor 产出代码后

**检查清单**：

| 序号 | 检查项 | 问题 |
|------|--------|------|
| 1 | 需求覆盖 | 所有需求都实现了吗？ |
| 2 | 代码质量 | 代码规范吗？有问题吗？ |
| 3 | 边界处理 | 边界情况处理了吗？ |
| 4 | 错误处理 | 异常情况处理了吗？ |

**质疑话术**：
```
【产出确认】

代码已产出，我检查后发现：

1. 已实现：<列表>
2. 可能问题：<列表>
3. 需要确认：<列表>

请您确认产出是否符合预期。
```

---

### 2.5 步骤5-6后：结果质疑

**时机**：qa/code-reviewer 产出报告后

**检查清单**：

| 序号 | 检查项 | 问题 |
|------|--------|------|
| 1 | 验证全面 | 所有验收标准都验证了吗？ |
| 2 | 问题修复 | 所有问题都修复了吗？ |
| 3 | 遗漏项 | 有没有遗漏的问题？ |

**质疑话术**：
```
【结果确认】

验证/审查报告已产出，我检查后发现：

1. 通过项：<列表>
2. 未通过项：<列表>
3. 需要修复：<列表>

请您确认结果是否可以接受。
```

## 三、质疑机制在流程中的位置

### 3.1 完整流程图（带质疑）

```
【步骤0】任务状态检测
    ↓
【步骤1】接收任务 + thinking-coach
    ↓
【质疑1】需求质疑 → 用户确认
    ↓
【步骤2】需求分析 + requirements-analyst
    ↓
【质疑2】需求完整性质疑 → 用户确认
    ↓
【步骤3】制定计划 + executor
    ↓
【质疑3】计划合理性质疑 → 用户确认
    ↓
【步骤4】执行任务 + executor（循环）
    ↓
【质疑4】产出质量质疑 → 用户确认
    ↓
【步骤5】功能验证 + qa
    ↓
【质疑5】验证结果质疑 → 用户确认
    ↓
【步骤6】代码审查 + code-reviewer
    ↓
【质疑6】审查结果质疑 → 用户确认
    ↓
【步骤8】完成
```

## 四、quietMode 和 unattendedMode 下的处理

### 4.1 quietMode = true（安静模式）

- 跳过所有质疑环节
- 自动标记"质疑通过"

### 4.2 unattendedMode = true（无人值守模式）

- 跳过所有质疑环节
- 自动标记"质疑通过"

### 4.3 普通模式

- 必须执行质疑环节
- 用户确认后才能继续
