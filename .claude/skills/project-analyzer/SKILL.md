# 项目分析专家

分析 Claude Code 插件项目，提取优点，与当前项目对比，从流程层面判断是否需要借鉴。

## 使用场景

- 想了解其他插件项目的设计思路
- 想找出当前项目的不足和优化点
- 想学习其他项目的最佳实践

## 分析框架

### 核心问题

分析时需要回答以下问题：

1. **目标项目有什么优点？** - 具体实现方式是什么
2. **当前项目实现了吗？** - 对比当前项目的实现情况
3. **从流程层面看需要添加吗？** - 判断是否真正需要，不是简单照搬
4. **如何结合当前项目特点？** - 不是复制，而是融合

### 当前项目流程

```
需求 → 规划 → 执行 → 审查 → 验证 → 完成
```

分析时需要结合这个流程，判断目标项目的优点属于哪个阶段。

## 分析流程

### Step 1: 克隆项目

```bash
# 克隆到临时目录
cd /tmp
git clone [仓库地址] project-analysis
cd project-analysis
```

### Step 2: 提取目标项目的优点

分析目标项目的核心优点，按以下维度分类：

| 维度 | 说明 |
|------|------|
| **功能亮点** | 独特的功能设计 |
| **工作流亮点** | 流程设计、自动化能力 |
| **技术亮点** | 脚本、Hooks、配置 |
| **文档亮点** | README、示例、模板 |

**分析方法：**
1. 读取 README.md 了解项目定位
2. 列出所有 Skills/Agents/Commands
3. 分析核心功能的实现方式
4. 提取独特的模式或设计

### Step 3: 当前项目对比

读取当前项目配置：

```bash
# 当前项目的 agents
ls agents/

# 当前项目的 skills
ls skills/

# 当前项目的 commands
ls commands/

# 当前项目的 hooks
ls hooks/
```

**对比维度：**

| 维度 | 分析项 |
|------|--------|
| **功能对比** | 目标项目的优点，当前项目是否有类似功能 |
| **流程对比** | 属于哪个阶段，当前流程是否覆盖 |
| **实现对比** | 实现方式差异，谁的方案更好 |
| **完整度对比** | 目标项目的完整度 vs 当前项目 |

### Step 4: 流程层面判断

**判断标准：**

| 判断项 | 是 | 否 |
|--------|----|----|
| **是否需要？** | 当前流程是否真的需要这个功能 | 盲目添加会增加复杂度 |
| **是否适配？** | 是否符合当前项目的定位和风格 | 照搬可能破坏一致性 |
| **成本收益** | 实现成本 vs 收益 | 高成本低收益不值得 |
| **可替代性** | 是否有其他方式实现 | 能用现有功能解决就不需要新功能 |

**决策矩阵：**

| 情况 | 决策 |
|------|------|
| 当前项目已实现 | ❌ 不需要添加 |
| 当前项目部分实现 | ⚠️ 可增强 |
| 需要添加但成本高 | 🔜 计划中 |
| 需要添加且成本低 | ✅ 建议添加 |
| 不适合当前项目 | ❌ 不建议添加 |

## 输出要求

1. **保存报告**到：`docs/plans/YYYY-MM-DD-[project-name]-analysis.md`
2. **不要修改**任何现有文件
3. **不要创建**任何新文件到目标项目
4. 只做**只读分析**

## 关键原则

1. **不是照抄** - 是判断是否需要，不是全部复制
2. **流程优先** - 从整体流程判断，不是从单个功能判断
3. **适配当前** - 结合当前项目特点，不是简单迁移
4. **成本收益** - 考虑实现成本和预期收益
