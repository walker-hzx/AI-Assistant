# 配置复盘

复盘在实践过程中使用 AI-Assistant 配置的情况，分析问题并提出改进建议。

## 使用方式

```
/config-retrospect
```

或者直接说"复盘"、"分析配置使用"

## 功能

复盘实践过程中的问题，分析原因并提出改进建议。

### 问题分类

| 类型 | 例子 | 说明 |
|------|------|------|
| **使用问题** | 需求不清晰就去做计划 | 用户使用方式不当 |
| **技能问题** | 代码无法运行却继续E2E测试 | 配置本身需要改进 |
| **流程问题** | 跳过了必要步骤 | 需要增强约束 |
| **环境问题** | 项目结构差异导致 | 需要适配 |

## 分析维度

### 1. 问题描述
- 发生了什么问题
- 在哪个环节发生的
- 具体表现是什么

### 2. 问题归因
- 是使用问题还是配置问题
- 如果是配置问题，具体是哪个技能/流程需要改进

### 3. 原因分析
- 为什么会这样
- 是否有更好的处理方式

### 4. 改进建议
- 如何改进配置
- 如何避免类似问题

## 输出格式

```markdown
## 配置复盘报告

### 问题列表
1. [问题1]
   - 类型：使用问题/技能问题
   - 环节：[哪个步骤]
   - 原因：[为什么]

2. [问题2]
   ...

### 改进建议
1. [改进建议1]
2. [改进建议2]

### 总结
[这次复盘的收获和后续行动计划]
```

## 复盘时机

建议在以下时机使用：
- 完成一个功能后
- 遇到明显问题后
- 项目阶段性完成后
- 定期（如每周）回顾

## 注意

- 复盘是为了改进，不是批评
- 区分使用问题和配置问题
- 提出具体的改进方向
