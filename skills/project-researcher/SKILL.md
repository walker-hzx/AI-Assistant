# 项目调研

## 概述

中途接手项目时，系统了解项目现状，评估代码质量，识别规范和封装，给出批判性评估。

**核心原则**：批判性继承 - 好的遵循，有问题的改进或重写。

---

## 使用场景

- 中途接手新项目
- 需要了解现有代码结构
- 需要识别项目规范和封装
- 需要评估是否值得复用现有代码

---

## 核心能力

### 1. 项目结构分析

```
分析维度：
- 目录结构是否清晰
- 模块划分是否合理
- 技术栈是什么
- 依赖关系如何
```

### 2. 编码规范识别

```
识别内容：
- 命名风格（变量、函数、文件）
- 注释规范
- 代码格式
- 常量定义方式
```

### 3. 封装评估

```
前端：
- 组件封装质量
- 状态管理方式
- API 调用封装
- 工具函数

后端：
- 工具函数封装
- 中间件设计
- 数据库操作封装
- 业务逻辑分层
```

### 4. 批判性评估

```
评估标准：
- 好的实践 → 遵循
- 一般 → 可改进
- 有问题 → 考虑重写
```

---

## 调研框架

### 【重要】调研前先了解背景

**在任何调研之前，必须先问清楚：**

1. **这个项目的性质？**
   - 新项目还是老项目？
   - 维护期还是开发期？

2. **调研的目标？**
   - 只是了解还是要长期维护？
   - 涉及哪些模块？

3. **有没有文档？**
   - README、技术文档、设计文档

---

### 调研步骤

#### 1. 整体结构扫描

```
【操作】
- 读取项目根目录结构
- 识别前端/后端目录
- 了解技术栈

【产出】
- 项目技术栈清单
- 目录结构概览
```

#### 2. 编码规范识别

```
【操作】
- 查找 2-3 个典型文件
- 分析命名风格
- 分析注释习惯
- 分析代码格式

【产出】
- 编码规范清单
- 示例代码
```

#### 3. 封装评估

```
【操作】
- 查找 components/、utils/、hooks/ 等目录
- 分析封装的粒度和质量
- 评估是否好用

【产出】
- 封装清单
- 质量评估（好/一般/差）
```

#### 4. 代码质量抽查

```
【操作】
- 抽查 3-5 个核心文件
- 检查常见问题

【产出】
- 问题清单
- 质量评级
```

## 关键原则

1. **批判性继承** - 不是全盘接受，也不是全盘否定
2. **区分主次** - 核心规范要遵循，次要可以灵活
3. **给出行动建议** - 知道问题后要告诉怎么做
4. **产出可操作** - 报告要能指导后续开发
