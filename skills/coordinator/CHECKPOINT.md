# 检查点定义

> 本文档定义任务执行过程中的检查点。根据任务类型选择需要的检查点。

---

## 检查点列表

### 检查点 0：任务状态检测

**检查内容**：
- [ ] 扫描 docs/coordinator/ 目录
- [ ] 判断是否有进行中的任务

---

### 检查点 1：调度记录创建

**检查内容**：
- [ ] `docs/coordinator/<task>-coordinator.md` 已创建
- [ ] 任务基本信息已填写
- [ ] 状态为"进行中"

---

### 检查点 2：需求分析（可选）

**检查内容**：
- [ ] `docs/intent/<task>-intent.md` 已创建（如果需要）
- [ ] 用户真实意图已明确
- [ ] 任务类型已判断

**此步骤可选**：简单任务可能不需要

---

### 检查点 3：制定计划（可选）

**检查内容**：
- [ ] `docs/plans/<task>-plan.md` 已创建（如果需要）
- [ ] 角色组合已确定
- [ ] 每个任务有执行者声明

**此步骤可选**：简单任务可能不需要

---

### 检查点 4：执行轮次完成

**检查内容**（每轮执行后）：
- [ ] 执行日志 `docs/execution/<task>-execution-N.md` 已创建
- [ ] Subagent 输出文档存在
- [ ] 任务状态已更新

---

### 检查点 5：功能验证

| 任务类型 | 要求 |
|----------|------|
| 调研任务 | 可选 |
| Bug修复 | 必做 |
| 功能开发 | 必做 |
| 复杂任务 | 必做 |
| 重构任务 | 必做 |
| 测试用例增强 | 必做 |
| UI/UX 优化 | 必做 |
| 安全审查 | 必做 |

**检查内容**：
- [ ] `docs/verification/<task>-verification.md` 已创建（如果需要）
- [ ] 功能验证通过

**此步骤可选**：仅调研任务可选，其他任务必做

---

### 检查点 6：代码审查

| 任务类型 | 要求 |
|----------|------|
| 调研任务 | 不需要 |
| Bug修复 | 可选 |
| 功能开发 | 必做 |
| 复杂任务 | 必做 |
| 重构任务 | 必做 |
| 测试用例增强 | 可选 |
| UI/UX 优化 | 必做 |
| 安全审查 | 必做 |

**检查内容**：
- [ ] `docs/reviews/<task>-review.md` 已创建（如果需要）
- [ ] 代码质量审查通过

**此步骤可选**：调研任务不需要，Bug修复和测试用例增强可选

---

### 检查点 7：质量评估（仅复杂任务）

**检查内容**：
- [ ] `docs/quality/<task>-summary.md` 已创建（仅复杂任务）
- [ ] 质量评估通过

**此步骤仅复杂任务需要**

---

### 检查点 8：任务完成

**检查内容**：
- [ ] 调度记录状态为"已完成"
- [ ] 所有必要步骤已完成

---

## 【核心】任务对照

| 类型与检查点任务类型 | 必做检查点 | 可选检查点 |
|----------|------------|-------------|
| 调研任务 | 0, 1, 4, 8 | 2, 3 |
| Bug修复 | 0, 1, 2, 4, 5, 8 | 3, 6 |
| 功能开发 | 0, 1, 2, 3, 4, 5, 6, 8 | 7 |
| 复杂任务 | 0, 1, 2, 3, 4, 5, 6, 7, 8 | - |
| 重构任务 | 0, 1, 2, 3, 4, 5, 6, 8 | 7 |
| 测试用例增强 | 0, 1, 2, 3, 4, 5, 8 | 6, 7 |
| UI/UX 优化 | 0, 1, 2, 4, 5, 6, 8 | 3, 7 |
| 安全审查 | 0, 1, 2, 4, 5, 6, 8 | 3, 7 |

### 质疑检查点（所有任务必做）

| 检查点 | 质疑类型 | 触发时机 | 输出文档 |
|--------|---------|----------|----------|
| 质疑1 | requirement | 步骤1完成后 | `docs/skeptics/<task>-requirement-skeptics.md` |
| 质疑2 | completeness | 步骤2完成后 | `docs/skeptics/<task>-completeness-skeptics.md` |
| 质疑3 | plan | 步骤3完成后 | `docs/skeptics/<task>-plan-skeptics.md` |
| 质疑4 | output | 步骤4完成后 | `docs/skeptics/<task>-output-skeptics.md` |
| 质疑5 | verification | 步骤5完成后 | `docs/skeptics/<task>-verification-skeptics.md` |
| 质疑6 | review | 步骤6完成后 | `docs/skeptics/<task>-review-skeptics.md` |

**注意**：quietMode=true 或 unattendedMode=true 时跳过质疑环节

### 触发条件说明

| 检查点 | 何时必做 | 何时可选 |
|--------|----------|----------|
| 需求分析 | 需求不清晰、复杂任务 | 需求明确、简单任务 |
| 制定计划 | 复杂任务、功能开发 | 简单任务、调研任务 |
| 功能验证 | 有功能产出的任务 | 纯调研任务 |
| 代码审查 | 功能开发、重构 | Bug修复、调研任务 |
| 质量评估 | 复杂任务 | 其他任务 |

```
- docs/coordinator/<task>-coordinator.md

```

## 【核心】内容质量检查

### 需求分析文档质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 功能目标 | 明确、具体、可测试 | 是否有明确的功能描述 |
| 业务规则 | 完整、无歧义 | 是否有遗漏的业务场景 |
| 验收标准 | 量化、可验证 | 是否有可测试的验收条件 |
| 风险识别 | 全面、有缓解措施 | 是否有风险点和应对方案 |

### 计划文档质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 任务拆分 | 2-5分钟粒度 | 每个任务是否可独立完成 |
| 依赖关系 | 清晰、无循环 | 任务间依赖是否正确 |
| 执行顺序 | 合理、最优 | 顺序是否符合逻辑 |
| 里程碑 | 明确、可交付 | 每个里程碑是否有产出 |

### 验证报告质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 测试覆盖 | 核心功能覆盖 | 是否有遗漏的关键功能 |
| 测试质量 | 可执行、有断言 | 测试是否真正验证功能 |
| 问题记录 | 完整、有截图 | 问题描述是否清晰 |

### 审查报告质量检查

| 检查项 | 标准 | 检查方法 |
|--------|------|----------|
| 问题识别 | 全面、准确 | 是否遗漏重要问题 |
| 严重程度 | 合理、正确 | 问题分级是否合理 |
| 修复建议 | 具体、可操作 | 建议是否可执行 |

## 失败处理原则

| 失败类型 | 处理方式 |
|----------|----------|
| 文档缺失 | 创建后重试 |
| 字段缺失 | 补充后重试 |
| 验证失败 | 修复后重试（最多 3 次） |
| 审查失败 | 修复后重试或接受风险 |
| 达到上限 | 通知用户，人工介入 |
