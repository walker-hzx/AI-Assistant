# 执行规范

> 本文档定义管家执行任务的完整流程规范

---

## 执行流程

### 完整流程图

```
【阶段 0】接收任务
    ↓
【阶段 1】创建调度记录
    ↓
【阶段 2】意图分析（可选）
    ↓
【阶段 3】制定计划
    ↓
【阶段 4】执行计划（循环）
    ↓
【阶段 5】验证
    ↓
【阶段 6】审查
    ↓
【阶段 7】完成
```

---

## 阶段详解

### 阶段 0：接收任务

**任务来源**：用户输入的需求

**处理**：
1. 理解用户需求
2. 判断任务类型（需求类/实现类/问题类/优化类）
3. 评估复杂度（简单/普通/复杂）

### 阶段 1：创建调度记录

**必须创建文档**：`docs/coordinator/<task>-coordinator.md`

**内容**：
```markdown
# 管家任务调度记录

**时间**：{YYYY-MM-DD HH:MM}
**任务**：{一句话描述}
**类型**：{简单/普通/复杂}
**状态**：进行中

## 任务概览
- 用户需求：{用户原始需求}
- 任务类型：{需求类/实现类/问题类/优化类}
- 复杂度：{简单/普通/复杂}

## 计划
{计划的角色组合和任务列表}

## 执行轮次
| 轮次 | 状态 | 角色 | 结果 |
|------|------|------|------|
| 1 | 待执行 | - | - |
```

### 阶段 2：意图分析（可选）

**调用角色**：brainstorming / requirement-analysis

**触发条件**：
- 用户需求不明确
- 需要多角度分析

**输出文档**：`docs/intent/<task>-intent.md`

### 阶段 3：制定计划

**调用角色**：writing-plans

**必须生成文档**：`docs/plans/<task>-plan.md`

**计划内容**：
- 角色组合
- 执行顺序
- 里程碑列表
- 每轮执行的子任务

### 阶段 4：执行计划（循环）

**核心机制**：闭环执行循环

```
【执行轮次 N】
1. 从计划中获取当前轮次的任务
2. 调度相应角色执行
3. 等待角色完成
4. 读取角色输出文档
5. 检查执行结果
6. 判断是否需要调整计划
    → 需要调整 → 更新计划 → 继续下一轮
    → 完成 → 进入验证阶段
```

**执行日志**：`docs/execution/<task>-execution-N.md`

### 阶段 5：验证

**调用角色**：verification

**输出文档**：`docs/verification/<task>-verification.md`

### 阶段 6：审查

**调用角色**：code-review

**输出文档**：`docs/reviews/<task>-review.md`

### 阶段 7：完成

**更新调度记录**：标记任务为"已完成"

---

## 执行轮次机制

### 什么是执行轮次

每次计划执行（可能包含多个角色调用）称为一个"轮次"。

### 轮次状态

| 状态 | 说明 |
|------|------|
| 待执行 | 计划中尚未执行 |
| 进行中 | 正在执行 |
| 已完成 | 执行成功，验证通过 |
| 需调整 | 执行有问题，需要更新计划 |
| 失败 | 执行失败，无法继续 |

### 轮次判断规则

```
每次角色执行完成后：
1. 读取输出文档
2. 检查执行结果
3. 判断是否完成计划中的任务
   → 完成 → 进入下一轮或验证阶段
   → 未完成 → 判断原因
       → 可修复 → 继续当前轮次
       → 需调整计划 → 更新计划，进入下一轮
       → 无法修复 → 标记失败，通知用户
```
