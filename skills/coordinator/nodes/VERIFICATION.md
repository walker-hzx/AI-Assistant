# Coordinator 验证节点

> 步骤5（功能验证）相关的详细节点

---

## 功能验证

> **验证代码是否正确实现需求**

**【强制】验证结果必须写入文档，且由 Subagent 创建**

### 执行

1. 验证功能是否按计划实现
2. 运行测试验证正确性
3. 如果有回归测试需求，运行回归测试
4. 使用 Task 调用 ai-assistant:qa 创建验证文档：`docs/verification/<task>-verification.md`

### 验证报告必须包含

- 验证方法
- 验证结果（通过/失败）
- 失败项及原因

### 门控检查

- 验证报告存在 → 读取验证结果
- 验证报告不存在 → 停止并提示用户
- 验证通过 → 进入下一步
- 验证未通过 → 修复后重新验证（最多 3 次）

---

## 量化验证

> **不只是"测了"，要"测够了"**

**【强制】功能开发、Bug修复必须完成量化验证**

### 触发条件

- 任务类型：功能开发、Bug修复、复杂任务
- 跳过条件：调研任务

### 量化验证清单

1. 覆盖率指标
   - [ ] 单元测试覆盖率 ≥ 80%
   - [ ] 核心逻辑覆盖率 = 100%
   - [ ] 边界条件覆盖率 ≥ 90%

2. 测试充分性
   - [ ] 正向用例覆盖
   - [ ] 负向用例覆盖
   - [ ] 边界用例覆盖
   - [ ] 异常用例覆盖

### 量化验证报告模板

```markdown
## 量化验证报告

### 覆盖率统计
- 单元测试覆盖率：85%
- 新增代码覆盖率：92%
- 边界条件覆盖率：95%

### 测试用例执行结果
- 通过：45
- 失败：0
- 跳过：3

### 质量门控
- [x] 覆盖率达标
- [x] 边界条件覆盖
- [x] 异常情况覆盖
```

### 执行

1. 使用 Task 调用 ai-assistant:qa 创建量化验证文档
2. 文档路径：`docs/verification/<task>-quantitative.md`
3. 读取量化结果，更新调度记录

### 门控检查

- 覆盖率达标 → 继续
- 覆盖率不达标 → 修复后重新验证

## 依赖影响分析

> **改这里会影响哪里？不破坏已有功能**

**【强制】功能开发、代码修改必须完成依赖影响分析**

### 触发条件

- 任务类型：功能开发、代码重构、复杂任务
- 跳过条件：调研任务、纯调研任务

### 依赖影响分析模板

#### 改动范围

本次改动文件：
- src/api/user.py
- src/services/auth.py

#### 影响分析

| 模块 | 影响类型 | 影响内容 | 验证方式 |
|------|----------|----------|----------|
| 登录流程 | 直接影响 | 登录逻辑变更 | 功能测试 |
| 用户中心 | 间接影响 | token校验 | 回归测试 |
| 管理员后台 | 无影响 | - | - |

#### 回归测试清单

- [ ] 登录功能回归
- [ ] 注册功能回归
- [ ] 登出功能回归

### 执行

1. 使用 Task 调用 ai-assistant:code-analysis 创建依赖影响分析文档
2. 文档路径：`docs/verification/<task>-impact.md`
3. 读取影响分析，更新调度记录

### 门控检查

- 检查文档存在 → 继续
- 检查文档不存在 → 停止并提示用户

---

## 验证结果质疑

> **验证验证报告的全面性**

1. 使用 Task 调用 ai-assistant:skeptics 进行验证结果质疑：
   - 任务描述：对验证报告进行质疑
   - 质疑类型：verification
   - 被质疑文档：docs/verification/<task>-verification.md
   - 输出：docs/skeptics/<task>-verification-skeptics.md
2. 读取质疑报告
3. 根据严重程度处理
