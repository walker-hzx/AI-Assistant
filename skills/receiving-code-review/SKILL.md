# 接收代码审查

## 概述

处理代码审查反馈时，需要技术严谨性和验证，而不是盲目执行。

**核心原则：** 质疑但不对抗，提供证据而非假设。

## 处理流程

### 1. 阅读反馈

- 理解每个问题
- 识别问题级别：CRITICAL / HIGH / MEDIUM / LOW

### 2. 分析问题

- 这个反馈正确吗？
- 是否有误解？
- 建议的修复合理吗？

### 3. 验证（关键）

**不要盲目执行反馈！**

- 如果是代码问题：阅读相关代码，验证问题是否存在
- 如果是安全问题：检查是否有漏洞
- 如果是风格问题：检查项目规范

### 4. 采取行动

| 情况 | 行动 |
|------|------|
| 反馈正确 | 执行修复 |
| 反馈有误 | 提供证据，说明理由 |
| 不确定 | 询问澄清 |
| 部分同意 | 同意部分，说明理由 |

## 反馈模板

### 接受反馈
```
同意。将按建议修复：

[修复内容]
```

### 拒绝反馈
```
我认为这个反馈需要重新考虑：

**理由：**
1. [理由 1]
2. [理由 2]

**证据：**
[代码/文档/测试结果]

**建议的替代方案：**
[如有]
```

### 询问澄清
```
需要澄清：

[具体问题]

感谢审查！
```

## 检查清单

处理反馈前：
- [ ] 理解每个反馈
- [ ] 验证问题是否真实存在
- [ ] 检查相关代码

处理反馈时：
- [ ] 如果正确，执行修复
- [ ] 如果有误，提供证据
- [ ] 如果不确定，询问澄清

处理反馈后：
- [ ] 更新代码
- [ ] 重新测试
- [ ] 感谢审查者
