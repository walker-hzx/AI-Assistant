# 编写实施计划

## 概述

编写全面的实现计划，假设工程师对你的代码库毫无了解。记录他们需要知道的一切：每个任务要接触哪些文件、代码、测试、如何测试。将整个计划作为小任务呈现。

**⚠️ 核心升级**：计划不只是"任务列表"，而是"经过深度思考的方案 + 任务列表"。在拆分任务之前，必须先完成方案设计和风险评估。

**开始时声明：** "我正在使用 writing-plans skill 来创建实现计划。"

---

## 【核心】计划阶段职责

**此阶段职责**：确定"怎么做"和"为什么这样做"

| 职责 | 说明 |
|------|------|
| **方案设计** | 多方案比较、选择最优路径、识别关键假设 |
| **风险评估** | 识别每种方案的风险点和应对策略 |
| 任务拆分 | 每个任务 2-5 分钟 |
| 依赖关系 | 哪些可以并行、哪些必须串行 |
| 验证方式 | 每个任务怎么验证 |
| 里程碑 | 阶段划分、检查点 |
| **功能完整性** | 确保每个功能点都有对应任务 |
| **假设清单** | 列出计划依赖的关键假设 |

**不在此阶段**：
- 边界条件具体实现 → 执行阶段
- 异常处理具体实现 → 执行阶段
- 需求层面确认 → brainstorming 阶段

**保存计划到：** `docs/plans/YYYY-MM-DD-<feature-name>.md`

---

## 【核心】创建计划流程

**AI 必须遵循的核心流程（6步）**：

```
1. 读取需求文档 + 环境扫描
   └─> docs/requirements/ 下的需求文档
   └─> 读取相关代码，理解现有实现

2. 问题建模
   └─> 这个需求的本质是什么？核心挑战是什么？
   └─> 涉及哪些模块？模块间关系？
   └─> 有哪些隐含约束和依赖？

3. 方案设计（MUST，不能跳过）
   └─> 至少提出 2 种实现方案
   └─> 分析每种方案的优劣（复杂度、风险、可维护性、扩展性）
   └─> 选择推荐方案并说明原因
   └─> 识别推荐方案的关键假设和风险点
   └─> 如果推荐方案失败的备选策略

4. 功能完整性检查
   └─> 确保每个功能点都有对应任务

5. 任务拆分（2-5分钟粒度）
   └─> 每个任务 = 1 个独立动作
   └─> 基于选定方案进行拆分

6. 确认 + 执行交接
   └─> 阶段门控确认提示
```

> **简化场景**：如果需求非常简单且方案显而易见，方案设计可以简化但不能完全跳过（至少说明为什么只有这一种合理方案）

## 【重要】输入输出文档规范

### 输入文档

**必须读取**：brainstorming 产出的需求文档

| 文档 | 位置 | 用途 |
|------|------|------|
| 需求文档 | `docs/requirements/YYYY-MM-DD-<feature>.md` | 获取"做什么"和"验收标准" |
| 需求总览 | `docs/requirements/README.md` | 获取当前有效需求列表 |

### 输出文档

**保存位置**：`docs/plans/YYYY-MM-DD-<feature>.md`

### 文档衔接

- **读取**：先读取 `docs/requirements/` 下的需求文档
- **产出**：计划文档保存到 `docs/plans/`
- **下一步**：计划制定完成，可进入执行阶段

## 小任务粒度

**每个步骤是一个动作（2-5 分钟）**：
- "编写失败的测试" - 步骤
- "运行它以确认失败" - 步骤
- "编写使测试通过的最小代码" - 步骤
- "运行测试确保通过" - 步骤
- "提交" - 步骤

## 迭代式计划结构

### 第零阶段：探索与方案设计（新增）
- **目标**：深入理解问题，选择最优实现路径
- **输出**：方案设计文档（问题分析 + 多方案比较 + 推荐方案 + 假设清单）
- **原则**：先理解问题空间，再设计解决方案；宁可多花时间探索，不要方向错误后返工

### 第一阶段：高层计划
- **目标**：明确要做什么
- **输出**：3-5 个主要阶段的粗略描述
- **原则**：先有整体框架，再逐个细化

### 第二阶段：里程碑细化
- **目标**：每个里程碑独立成计划
- **输出**：每个里程碑的详细任务列表
- **原则**：可以逐个细化，不必一次性完成所有细节

### 第三阶段：任务细化
- **目标**：每个任务 = 1 个独立步骤
- **最细粒度**：2-5 分钟可完成
- **原则**：越小越容易验证和调整

## 计划文档头部

**每个计划必须以此头部开头**：

```markdown
# [功能名称] 实现计划

> **For Claude:** 必需子技能：使用 executing-plans 逐个任务实现此计划。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句话描述方法]

**技术栈：** [根据项目实际情况]

---

**关联需求：**
- 直接相关：[需求名称]（必须满足）
- 间接相关：[需求名称]（需注意）
- 潜在冲突：[如有]

**方案选型：** [简述为什么选择了当前方案，而非其他方案]

**关键假设：**
- [待验证] 假设 1：...
- [待验证] 假设 2：...

**风险应对：**
- 风险 1 → 应对策略
- 假设失败 → 备选方案
```

**【新增】界面/组件设计任务**（涉及前端功能时包含）：

```markdown
### 任务 N：界面/组件设计

**界面设计**：
- [ ] 页面布局结构
- [ ] 交互流程（点击→反馈）
- [ ] 状态展示（加载/成功/失败）

**组件划分**：
- [ ] 需要哪些组件
- [ ] 组件职责划分
```

## 任务结构

```markdown
### 任务 N：[组件名称]

**依赖：**
- 前置任务：任务X, 任务Y（必须先完成）
- 可并行任务：任务A, 任务B（可并行执行）

**文件：**
- 创建：`exact/path/to/file.ts`
- 修改：`exact/path/to/existing.ts:123-145`
- 测试：`tests/exact/path/to/test.ts`

**步骤 1：编写失败的测试**
```typescript
function test_specific_behavior() {
  const result = function(input)
  expect(result).toBe(expected)
}
```

**步骤 2：运行测试验证失败**
运行：`npm test tests/path/test.ts`
预期：FAIL with "function not defined"

**步骤 3：编写最小实现**
```typescript
function function(input) {
  return expected
}
```

**步骤 4：运行测试验证通过**
运行：`npm test tests/path/test.ts`
预期：PASS

**步骤 5：提交**
```bash
git add tests/path/test.ts src/path/file.ts
git commit -m "feat: add specific feature"
```
```

## 功能完整性检查

对于每类功能，AI 主动思考常见功能点：

| 功能类型 | 常见功能点 |
|----------|-----------|
| 管理类（CRUD） | 列表、新增、编辑、删除、分页、搜索 |
| 配置类 | 查看、编辑、校验、保存 |
| 报表类 | 列表、分页、筛选、导出 |
| 权限类 | 列表、授权、撤销 |

**常见遗漏检查**：
- [ ] 列表分页
- [ ] 搜索/筛选
- [ ] 表单验证
- [ ] 删除确认
- [ ] 批量操作
- [ ] 状态变更（启用/禁用）

## 记住

- 始终使用精确的文件路径
- 完整代码在计划中（不是"添加验证"）
- 精确命令带预期输出
- DRY, YAGNI, TDD, 频繁提交

## 计划调整

### 何时调整
- 里程碑完成后对照原始需求检查
- 发现计划遗漏的步骤
- 需求有变化
- 发现新的依赖或风险

### 调整流程
1. 暂停当前进度
2. 分析需要调整的部分
3. 与用户确认调整方案
4. 更新计划后继续执行

## 【阶段门控】确认提示 + 执行交接

```
计划已完成！

📋 已确认内容：
- [ ] 任务拆分（每个任务 2-5 分钟）
- [ ] 依赖关系（可并行/串行）
- [ ] 验证方式（每个任务怎么验证）
- [ ] 里程碑（阶段划分）

⚠️ 需要您确认：
1. 计划理解是否正确？
2. 任务拆分是否合理？

✅ 计划制定完成，可进入执行阶段

> **注意**：计划完成后向用户确认，不要直接进入执行阶段
> 由用户决定何时开始执行

---

### 执行选择

**三个执行选项**：

1. **子代理驱动（本会话）** - 每个任务分配新的子代理，串行执行，任务间审查

2. **混合执行（本会话）** - 分析任务依赖关系，可并行任务同时执行

3. **并行会话（单独）** - 在新会话中打开 executing-plans

**推荐**：
- 有可并行任务 → 混合执行
- 全部串行 → 子代理驱动
- 想单独执行 → 并行会话
