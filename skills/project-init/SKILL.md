# 项目初始化

> 在一个新项目中首次使用 AI-Assistant 时运行。扫描项目并生成开箱即用的全局配置。

---

## 使用时机

- **新项目首次接入**：从没用过 AI-Assistant 的项目
- **项目接手**：接手已有项目，需要快速建立上下文
- **重置配置**：现有配置已经过时，需要重新生成

---

## 工作流程

### Step 1：扫描项目

并行读取项目关键文件，识别项目特征：

**技术栈识别**：
- 检查 `package.json` → 前端框架、主要依赖
- 检查 `pyproject.toml` / `requirements.txt` / `setup.py` → Python 后端
- 检查 `go.mod` → Go
- 检查 `Cargo.toml` → Rust
- 检查 `pom.xml` / `build.gradle` → Java

**目录结构分析**：
- 列出根目录下的主要文件夹
- 识别 `src/`、`app/`、`lib/` 等源码目录结构
- 识别测试目录：`tests/`、`__tests__/`、`spec/` 等
- 识别配置文件：`.env.example`、`docker-compose.yml` 等

**已有文档扫描**：
- 检查是否已有 `README.md`（提取项目概述）
- 检查是否已有 `CLAUDE.md`（避免覆盖现有配置）
- 检查是否已有 `docs/蓝图.md`（避免覆盖）

### Step 2：生成草稿

**⚠️ 不直接写入文件，先展示草稿供用户确认。**

生成以下文件的草稿内容：

#### 草稿 1：`CLAUDE.md`（项目根目录）

包含：
- 项目概述（从 README 或扫描结果提取）
- 技术栈（自动检测到的）
- 目录结构说明（精简版，只写关键目录）
- 可用命令参考（固定内容，来自 AI-Assistant 插件）
- 工作流指南（S/M/L 档）
- 项目特有的开发规范（如有 eslint/prettier 配置则提及）

#### 草稿 2：`docs/蓝图.md`

从模板出发，填入已知信息：
- 项目名称（从 package.json 的 name 或目录名）
- 技术栈（已检测）
- 功能清单（如有 README 描述功能则预填，否则留空）

#### 草稿 3：`docs/requirements/README.md`

需求追踪表初始化：
- 如果已有需求可识别则预填
- 否则生成空模板

### Step 3：展示草稿并确认

以清晰的格式展示三份草稿，询问用户：

```
已扫描项目。检测到：
- 技术栈：[检测结果]
- 主要目录：[列表]
- 已有文档：[发现的文档]

准备生成以下文件：
1. CLAUDE.md（项目根目录）
2. docs/蓝图.md
3. docs/requirements/README.md

[展示草稿内容]

---
是否确认生成？可以先修改草稿内容。
```

### Step 4：写入文件

用户确认后：

1. **写入 `CLAUDE.md`**：
   - 如果已存在 → 询问是否覆盖或合并
   - 如果不存在 → 直接创建

2. **创建 `docs/` 目录结构**：
   ```
   docs/
   ├── 蓝图.md          ← 项目状态总览
   ├── requirements/
   │   └── README.md    ← 需求追踪
   ├── plans/           ← /plan 命令产出（目录）
   └── execution/       ← /executing-plans 产出（目录）
   ```

3. **写入 `docs/蓝图.md`**：
   - 如果已存在 → 跳过，告知用户

4. **写入 `docs/requirements/README.md`**：
   - 如果已存在 → 跳过，告知用户

### Step 5：完成提示

```
✅ 项目初始化完成！

生成的文件：
- CLAUDE.md
- docs/蓝图.md
- docs/requirements/README.md

后续建议：
1. 检查 CLAUDE.md，补充项目特有的规范和约定
2. 填写 docs/蓝图.md 的功能清单
3. 开始使用 /discuss 进行需求讨论

可用命令：
/discuss - 需求讨论
/plan - 制定计划
/implement - 实现功能
```

## 生成文件内容规范

### CLAUDE.md 模板

```markdown
# [项目名称]

> Claude Code 项目配置

## 项目概述

[从 README 提取或简短描述]

## 技术栈

[自动检测填写]

## 目录结构

[关键目录说明]

## 开发流程

直达模式: 用户 → /命令 → 对应 Agent 或 Skill → 完成

| 任务规模 | 使用方式 |
|----------|----------|
| 小任务 | 直接对话或单个命令 |
| 中任务 | 命令 + TodoWrite 跟踪 |
| 大任务 | /plan 制定计划 → /executing-plans 分步执行 |

## 可用命令

- `/discuss` - 需求讨论和头脑风暴
- `/plan` - 制定实施计划
- `/executing-plans` - 执行计划
- `/implement` - 实现功能
- `/refactor` - 重构代码
- `/review` - 代码审查
- `/security-review` - 安全审查
- `/debugging` - 调试定位
- `/verification` - 功能验证
- `/test-planner` - 测试设计
- `/analyze` - 代码/方案分析
- `/docs` - 编写文档
- `/sync-progress` - 同步蓝图和需求状态

## 质量标准

- 测试覆盖率: 80%+
- 代码审查: 所有重要变更

## 项目约定

[项目特有的约定，初始化时留空，开发过程中补充]
```

### docs/requirements/README.md 模板

```markdown
# 需求追踪

> 所有需求的单一事实来源。制定计划前必须读取此文件。

## 状态说明

- 📋 待开发：已确认但未开始

- ✅ 已完成：开发验证完成

## 当前需求

| 需求 | 优先级 | 状态 | 关联计划 | 备注 |
|------|--------|------|----------|------|
| | P0 | 📋 待开发 | | |

## 已作废需求

| 需求 | 作废原因 | 作废时间 |
|------|----------|----------|
| | | |
```

---

## 注意事项

- **不覆盖已有配置**：如果 CLAUDE.md 已存在且内容完整，询问用户而非直接覆盖
- **保留用户自定义**：合并而不是替换时，保留用户已有的约定和规范
- **适配不同技术栈**：生成的 CLAUDE.md 根据检测到的技术栈调整内容
- **最小化原则**：生成的配置不要太长，留给用户自己填充项目特有内容
