# 请求代码审查

## 概述

派遣 code-reviewer 子代理进行代码审查，在问题蔓延之前捕获它们。

**核心原则：** 尽早审查，频繁审查。

## 【重要】提交前自检

在调用 code-reviewer agent 之前，先进行自我审查：

### 为什么需要自检

- 直接交给 agent → agent 发现很多问题 → 你来回改 → 效率低
- 先自检一遍 → 减少低级问题 → agent 专注于更高价值的问题

### 自检步骤

**1. 快速扫描代码**

检查以下常见问题：
- [ ] 是否有明显的语法错误
- [ ] 是否有 console.log 或调试代码
- [ ] 是否有未使用的变量/导入
- [ ] 是否有 TODO 待完成

**2. 运行本地验证**

- [ ] 测试是否通过
- [ ] Lint 检查是否通过
- [ ] 类型检查是否通过
- [ ] 构建是否成功

**3. 主动分析问题**

思考以下问题：
- 这段代码可能有什么隐患？
- 边界条件都处理了吗？
- 错误情况都考虑了吗？
- 有没有安全漏洞？

**4. 筛选重点**

基于自检结果，确定重点审查内容：
- 如果自检发现很多问题 → 先修复，再请求审查
- 如果自检基本没问题 → 可以请求审查

## 何时请求审查

**必须审查：**
- 完成主要功能后
- 合并到 main 分支前
- 提交 PR 前

**可选但有价值：**
- 遇到困难时（获得新视角）
- 重构前（基线检查）
- 修复复杂 bug 后

## 如何请求

**1. 获取 git 信息：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)
HEAD_SHA=$(git rev-parse HEAD)
```

**2. 调用 code-reviewer agent：**

```bash
Task: code-reviewer
描述: 审查刚完成的用户登录功能
```

## 审查内容

### 必须检查
- [ ] 功能正确性
- [ ] 代码安全性
- [ ] 错误处理
- [ ] 测试覆盖

### 建议检查
- [ ] 代码可读性
- [ ] 命名规范
- [ ] 性能考虑
- [ ] 文档更新
