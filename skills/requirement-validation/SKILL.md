# 需求验证

## 概述

确认需求是否完整、清晰、可执行。

**核心理念**：检查需求中的遗漏和矛盾，确保需求可以进入计划阶段。

---

## 验证清单

### 1. 目标验证

| 检查项 | 状态 |
|--------|------|
| 核心问题已明确 | [ ] |
| 用户场景已明确 | [ ] |
| 成功标准已明确 | [ ] |

### 2. 范围验证

| 检查项 | 状态 |
|--------|------|
| 明确要做的功能 | [ ] |
| 明确不做的功能 | [ ] |
| 明确边界条件 | [ ] |

### 3. 依赖验证

| 检查项 | 状态 |
|--------|------|
| 依赖的现有模块已识别 | [ ] |
| 外部依赖已识别 | [ ] |
| 依赖关系清晰 | [ ] |

### 4. 风险验证

| 检查项 | 状态 |
|--------|------|
| 技术风险已识别 | [ ] |
| 依赖风险已识别 | [ ] |
| 缓解措施已考虑 | [ ] |

### 5. 验收标准

| 检查项 | 状态 |
|--------|------|
| 成功标准已明确 | [ ] |
| 失败标准已明确 | [ ] |
| 可测试的验收条件 | [ ] |

---

## 工作流程

1. **逐项检查**
   - 对照验证清单逐项检查
   - 标记已完成和未完成项

2. **识别问题**
   - 找出未完成的项
   - 识别需求中的矛盾

3. **补充或确认**
   - 能补充的及时补充
   - 无法补充的与用户确认

4. **给出结论**
   - 验证通过 → 可以进入计划阶段
   - 验证不通过 → 列出需要补充的内容

---

## 适用场景

- 需求理解和分析完成后
- 需要确认需求是否可以进入计划阶段
- 用户要求验证需求完整性

---

## 注意事项

- 不要跳过任何检查项
- 验证不通过不要进入计划阶段
- 明确列出需要补充的内容
