# 文档编写

> 从代码和上下文生成准确、有用的文档。

---

## 文档类型识别

| 类型 | 适用场景 | 关键内容 |
|------|---------|---------|
| README | 项目/模块说明 | 是什么、怎么用、怎么开发 |
| API 文档 | 端点说明 | 路径、参数、返回值、示例 |
| 架构文档 | 系统设计 | 模块关系、数据流、设计决策 |
| 变更记录 | 版本发布 | 新增、修复、破坏性变更 |
| 使用指南 | How-to | 步骤、示例、常见问题 |

---

## 编写流程

### 1. 确定文档目标

- **读者是谁**？（新人、使用者、维护者）
- **文档类型**？（参考上表选择）
- **覆盖范围**？（整个项目、单个模块、单个功能）

### 2. 收集信息

- 读取相关代码，提取核心逻辑
- 查看现有文档（避免重复或矛盾）
- 查看测试用例（了解预期行为）
- 查看 git log（了解变更历史）

### 3. 组织内容

按文档类型使用对应模板（见下方）。

### 4. 验证准确性

- **代码对照** — 文档描述与代码实际行为一致
- **示例可运行** — 代码示例是可执行的，不是伪代码
- **链接有效** — 引用的文件/路径存在

## 文档模板

### README 模板

```markdown
# 项目/模块名

> 一句话说明是什么

## 快速开始

（最简安装和使用步骤）

## 功能

（核心功能列表）

## 目录结构

（关键目录说明）

## 开发

（开发环境搭建、常用命令）

## 贡献

（贡献指南，如果需要）
```

### API 端点模板

```markdown
## POST /api/v1/items

创建新项目。

### 请求

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 项目名称 |

### 响应

| 字段 | 类型 | 说明 |
|------|------|------|
| id | number | 项目 ID |
| name | string | 项目名称 |

### 示例

请求：（具体示例）
响应：（具体示例）
```

### 架构文档模板

```markdown
# 模块名 - 架构说明

## 概述

（模块职责，一段话）

## 模块关系

（依赖了谁，被谁依赖）

## 数据流

（数据怎么流转的）

## 设计决策

（为什么这样设计，考虑过什么替代方案）
```

## 质量标准

| 维度 | 标准 |
|------|------|
| 准确性 | 与代码实际行为一致，无过时信息 |
| 完整性 | 核心功能全部覆盖，无关键遗漏 |
| 可读性 | 新人能看懂，术语有解释 |
| 简洁性 | 不写废话，不重复，信息密度高 |
| 可维护 | 结构清晰，后续容易更新 |

---

## 注意事项

- **不写自动生成能看出的信息** — 比如"这个函数接收一个参数 x"，代码签名已经说了
- **写"为什么"而非"是什么"** — 解释设计决策比描述代码更有价值
- **示例胜过描述** — 一个好例子比三段解释更清楚
- **保持更新** — 如果代码改了但文档没改，文档就是有害的

---

## 输出标准

文档编写完成后汇报：
- 生成了什么类型的文档
- 文档文件位置
- 覆盖了哪些内容
- 需要后续维护的部分（如果有）
