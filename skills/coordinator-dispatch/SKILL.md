# 任务派发

## 职责

按照 coordinator-planning 制定的方案，调度角色执行，监控进度，收集结果，更新执行日志。

**定位**：管家的"执行"环节

**核心原则**：
- 只负责"执行派发"（按方案调用角色）
- 不负责"决定派发给谁"（那是 coordinator-planning 的职责）

---

## 派发机制

### 【重要】必须使用 Skill

**必须使用 Skill 工具调用，不能使用 Task**。

原因：
- Task 调用 subagent 会创建独立上下文，执行完后**不会自动回到 coordinator**
- Skill 在同一个上下文运行，coordinator 可以控制**整个流程**

```
使用 Skill 工具调用：
- Skill: ai-assistant:brainstorming
- Skill: ai-assistant:writing-plans

示例：
/brainstorming
/executing-plans
```

### 不允许使用 Task

❌ **禁止使用 Task 工具调用其他 Agent**：
```
Task(subagent_type="ai-assistant:planner", prompt="...")  # 禁止
```

因为 Task 会创建独立子上下文，无法控制整个流程。

---

## 【重要】任务转交机制

> **核心原则**：所有其他角色不能直接调用其他角色，只能向 Coordinator 报告。

### 处理其他角色的协助请求

**当其他角色报告需要协助时**：

```
【收到协助请求】

角色：[角色名]
请求：需要 [XXX 角色] 协助
原因：[为什么需要]
问题：[具体问题]
```

**Coordinator 的决策流程**：
1. 评估请求是否合理
2. 决定是否调度其他角色
3. 如果调度，选择合适的角色
4. 告知原角色如何继续

**决策输出**：
```
【Coordinator 决策】

请求来源：[角色名]
请求内容：需要 [XXX 角色] 协助
决策结果：[批准/不批准]
调度角色：[如果批准，调度谁]
原角色后续：[如何继续执行]
```

## 执行流程

### 步骤 1：开始执行

**向用户展示**：
```
## 开始执行

**任务**：[任务名称]
**当前阶段**：里程碑 1 / 总 N 个里程碑
**调度**：[Skill/Agent 名称]
```

### 步骤 2：执行监控

**每个阶段执行中**：
- 调用 Skill 等待其完成
- Skill 完成后，coordinator **必须继续执行后续步骤**
- 记录执行状态
- 遇到问题及时处理

**【重要】Skill 执行完后必须继续**：
```
Skill 执行完成
    ↓
【管家】正在收集结果...
    ↓
【管家】正在生成文档...
    ↓
继续下一个里程碑或结束
```

❌ **错误做法**：Skill 执行完就结束，不生成文档
✅ **正确做法**：Skill 执行完后，继续收集结果、生成文档

## 【重要】执行循环机制

### 背景

对于复杂任务，可能需要多次修复才能完成。引入循环机制可以实现：
- **自动修复**：验证失败后自动调用修复
- **迭代改进**：直到验证通过或达到上限
- **风险控制**：最大循环次数防止无限循环

### 循环流程

```
[executor] 执行任务
    ↓
[qa/verification] 验证结果
    ↓
┌─────────────────────────────────────┐
│ 验证通过? → 继续下一个任务           │
└─────────────────────────────────────┘
    ↓ 否
┌─────────────────────────────────────┐
│ 循环次数 >= 3?                      │
│   → 退出，提示用户手动处理          │
└─────────────────────────────────────┘
    ↓ 否
[debugger] 修复问题
    ↓
回到 [qa/verification] 验证
```

### 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 最大循环次数 | 3 | 达到上限强制退出 |
| 每次循环询问 | 否 | 可配置是否每次询问用户 |

### 退出条件

| 条件 | 说明 |
|------|------|
| 验证通过 | qa/verification 返回成功 |
| 达到上限 | 循环 3 次后强制退出 |
| 无法修复 | debugger 判断无法自动修复 |
| 用户确认 | 用户手动确认完成 |

### 使用场景

**适用场景**：
- 复杂任务（有多个子任务）
- 预期需要多次修复
- 明确的验证标准

**不适用场景**：
- 简单任务（1 次性完成）
- 需求不明确（先解决需求问题）
- 用户需要全程掌控

### 风险控制

| 风险 | 防护措施 |
|------|---------|
| 无限循环 | 最大 3 次强制退出 |
| 修复变糟 | 每次修复后验证，记录变更 |
| 需求问题 | 循环 2 次后提示可能是需求问题 |
| 用户疲劳 | 每次循环后询问是否继续 |

### 简化版（推荐）

如果不需要完整循环，可以使用简化版：

```
执行 → 验证 → 失败? → 修复一次 → 验证
                                ↓
                              成功 → 继续
```

简化版只尝试修复 1 次，适合大多数场景。

## 【重要】文档输出规范

> 参考：[角色输出标准](../../docs/standards/role-output-standard.md)

### 必须生成的文档

| 文档 | 路径 | 生成时机 |
|------|------|---------|
| **管家调度记录** | `docs/plans/YYYY-MM-DD-<task>-coordinator.md` | 开始执行前 |
| **执行日志** | `docs/plans/YYYY-MM-DD-<task>-execution-log.md` | 执行过程中持续更新 |

### 管家调度记录内容

```markdown
# 管家任务记录

**时间**：{YYYY-MM-DD HH:MM}
**任务**：{一句话描述}
**类型**：{简单/普通/复杂}

## 意图分析
- 用户真实意图：{从 coordinator-intent 获取}
- 任务类型：{bug修复/功能开发/优化/其他}
- 复杂度判断：{简单/普通/复杂}

## 方案规划
- 选择的角色：{从 coordinator-planning 获取}
- 执行顺序：{阶段列表}
- 里程碑：{每个阶段的检查点}

## 任务分派
| 阶段 | 角色 | 任务 | 状态 |
|------|------|------|------|
| 1 | requirements-analyst | 需求分析 | 待执行 |
| 2 | planner | 制定计划 | 待执行 |
| 3 | executor | 代码实现 | 待执行 |
| ... | ... | ... | ... |

## 执行日志
{执行过程中持续更新}

## 结果汇总
{任务完成后汇总}
```

### 输出示例

```
【管家】任务已分配，开始执行

【调度记录】
📋 任务：用户登录功能
📊 类型：功能开发
👥 角色：requirements-analyst → planner → executor → qa
📝 文档：docs/plans/2026-02-26-user-login-coordinator.md

[阶段 1] requirements-analyst 正在分析需求...
```

---

**问题处理**：
```
发现问题了：
- 问题描述：[是什么]
- 影响范围：[多大]
- 解决方案：[如何处理]

需要调整方案吗？(Y/N)
```

### 步骤 3：结果收集

**每个阶段完成后**：
- 收集产出物
- 验证是否达标
- 记录完成状态

### 步骤 4：阶段衔接

**阶段间检查**：
- 当前阶段是否完成？
- 产出是否满足要求？
- 可以进入下一阶段吗？

## 日志更新

### 更新时机

| 时机 | 更新内容 |
|------|---------|
| 阶段开始 | 开始时间、当前状态 |
| 阶段完成 | 完成时间、产出物 |
| 问题发现 | 问题描述、影响 |
| 方案调整 | 调整内容、理由 |

### 日志格式

```markdown
## 执行日志

### 里程碑 1：[名称]

**状态**：进行中 / 已完成 / 有问题

**开始时间**：YYYY-MM-DD HH:MM

**执行记录**：
- HH:MM：开始执行
- HH:MM：完成步骤 1
- HH:MM：完成步骤 2

**产出物**：
- [文件/功能描述]

**遇到的问题**（如有）：
- 问题描述

### 里程碑 2：[名称]

**状态**：待开始 / 进行中
```

## 完成处理

### 任务完成时

**收集最终结果**：
- 所有产出物
- 执行摘要
- 遇到的问题及解决

**更新日志为完成状态**：
```markdown
## 执行完成

**完成时间**：YYYY-MM-DD HH:MM
**总耗时**：X 分钟

**最终结果**：
- [产出物列表]

**执行摘要**：
- 总阶段数：N
- 按时完成：X
- 有问题：Y
- 调整次数：Z
```
