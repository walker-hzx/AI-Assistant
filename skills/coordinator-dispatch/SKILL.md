# 任务派发

## 职责

按照 coordinator-planning 制定的方案，调度角色执行，监控进度，收集结果，更新执行日志。

**定位**：管家的"执行"环节

**核心原则**：
- 只负责"执行派发"（按方案调用角色）
- 不负责"决定派发给谁"（那是 coordinator-planning 的职责）

---

## 派发机制

### 【重要】必须使用 Skill

**必须使用 Skill 工具调用，不能使用 Task**。

原因：
- Task 调用 subagent 会创建独立上下文，执行完后**不会自动回到 coordinator**
- Skill 在同一个上下文运行，coordinator 可以控制**整个流程**

```
使用 Skill 工具调用：
- Skill: ai-assistant:brainstorming
- Skill: ai-assistant:writing-plans

示例：
/brainstorming
/executing-plans
```

### 不允许使用 Task

❌ **禁止使用 Task 工具调用其他 Agent**：
```
Task(subagent_type="ai-assistant:planner", prompt="...")  # 禁止
```

因为 Task 会创建独立子上下文，无法控制整个流程。

## 执行流程

### 步骤 1：开始执行

**向用户展示**：
```
## 开始执行

**任务**：[任务名称]
**当前阶段**：里程碑 1 / 总 N 个里程碑
**调度**：[Skill/Agent 名称]
```

### 步骤 2：执行监控

**每个阶段执行中**：
- 调用 Skill 等待其完成
- Skill 完成后，coordinator **必须继续执行后续步骤**
- 记录执行状态
- 遇到问题及时处理

**【重要】Skill 执行完后必须继续**：
```
Skill 执行完成
    ↓
【管家】正在收集结果...
    ↓
【管家】正在生成文档...
    ↓
继续下一个里程碑或结束
```

❌ **错误做法**：Skill 执行完就结束，不生成文档
✅ **正确做法**：Skill 执行完后，继续收集结果、生成文档

## 【重要】文档输出规范

### 必须生成的文档

| 文档 | 路径 | 生成时机 |
|------|------|---------|
| **管家调度记录** | `docs/plans/YYYY-MM-DD-<task>-coordinator.md` | 开始执行前 |
| **执行日志** | `docs/plans/YYYY-MM-DD-<task>-execution-log.md` | 执行过程中持续更新 |

### 管家调度记录内容

```markdown
# 管家任务记录

**时间**：{YYYY-MM-DD HH:MM}
**任务**：{一句话描述}
**类型**：{简单/普通/复杂}

## 意图分析
- 用户真实意图：{从 coordinator-intent 获取}
- 任务类型：{bug修复/功能开发/优化/其他}
- 复杂度判断：{简单/普通/复杂}

## 方案规划
- 选择的角色：{从 coordinator-planning 获取}
- 执行顺序：{阶段列表}
- 里程碑：{每个阶段的检查点}

## 任务分派
| 阶段 | 角色 | 任务 | 状态 |
|------|------|------|------|
| 1 | requirements-analyst | 需求分析 | 待执行 |
| 2 | planner | 制定计划 | 待执行 |
| 3 | executor | 代码实现 | 待执行 |
| ... | ... | ... | ... |

## 执行日志
{执行过程中持续更新}

## 结果汇总
{任务完成后汇总}
```

### 输出示例

```
【管家】任务已分配，开始执行

【调度记录】
📋 任务：用户登录功能
📊 类型：功能开发
👥 角色：requirements-analyst → planner → executor → qa
📝 文档：docs/plans/2026-02-26-user-login-coordinator.md

[阶段 1] requirements-analyst 正在分析需求...
```

---

**问题处理**：
```
发现问题了：
- 问题描述：[是什么]
- 影响范围：[多大]
- 解决方案：[如何处理]

需要调整方案吗？(Y/N)
```

### 步骤 3：结果收集

**每个阶段完成后**：
- 收集产出物
- 验证是否达标
- 记录完成状态

### 步骤 4：阶段衔接

**阶段间检查**：
- 当前阶段是否完成？
- 产出是否满足要求？
- 可以进入下一阶段吗？

## 日志更新

### 更新时机

| 时机 | 更新内容 |
|------|---------|
| 阶段开始 | 开始时间、当前状态 |
| 阶段完成 | 完成时间、产出物 |
| 问题发现 | 问题描述、影响 |
| 方案调整 | 调整内容、理由 |

### 日志格式

```markdown
## 执行日志

### 里程碑 1：[名称]

**状态**：进行中 / 已完成 / 有问题

**开始时间**：YYYY-MM-DD HH:MM

**执行记录**：
- HH:MM：开始执行
- HH:MM：完成步骤 1
- HH:MM：完成步骤 2

**产出物**：
- [文件/功能描述]

**遇到的问题**（如有）：
- 问题描述

### 里程碑 2：[名称]

**状态**：待开始 / 进行中
```

## 进度汇报

### 向用户汇报的格式

```
## 执行进度

**任务**：[名称]
**当前阶段**：里程碑 X / 总 Y 个
**进度**：XX%

### 已完成
- [x] 里程碑 1：完成
- [x] 里程碑 2：完成

### 进行中
- [ ] 里程碑 3：正在执行...

### 待处理
- [ ] 里程碑 4

**预计剩余时间**：约 Z 分钟
```

---

## 完成处理

### 任务完成时

**收集最终结果**：
- 所有产出物
- 执行摘要
- 遇到的问题及解决

**更新日志为完成状态**：
```markdown
## 执行完成

**完成时间**：YYYY-MM-DD HH:MM
**总耗时**：X 分钟

**最终结果**：
- [产出物列表]

**执行摘要**：
- 总阶段数：N
- 按时完成：X
- 有问题：Y
- 调整次数：Z
```
