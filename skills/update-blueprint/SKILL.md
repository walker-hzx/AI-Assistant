# 项目蓝图更新助手

## 使用场景

- 需求明确后
- 技术方案确定后
- 完成一个功能后
- 计划调整后

## 工作流程

### Step 1: 确定更新时机

询问用户现在是什么阶段：

1. 需求明确 → 添加功能描述
2. 技术方案确定 → 添加数据库设计、API 接口
3. 完成功能 → 更新 API 状态为"已完成"
4. 计划调整 → 更新任务状态

### Step 2: 收集信息

根据阶段，收集需要更新的信息：

**需求明确后**：
- 功能名称
- 核心描述
- 技术选型

**技术方案后**：
- 数据库表结构
- API 接口定义
- 核心类型

**功能完成后**：
- 完成的 API 状态
- 新增的数据库表
- 新增的类型

### Step 3: 更新蓝图

找到 `docs/蓝图.md` 文件，更新相应内容：

1. **功能清单**：添加或更新功能状态
2. **数据库设计**：添加新表结构
3. **API 接口**：添加或更新接口状态
4. **前端类型**：添加新的类型定义

### Step 4: 确认

展示更新后的相关内容，确认无误。

## 蓝图内容模板

```markdown
# 项目蓝图：项目名称

## 1. 项目概述
- 核心功能：
- 技术栈：根据项目实际情况填写

## 2. 功能清单
| 功能 | 描述 | 状态 |
|------|------|------|
| 用户登录 | 邮箱密码登录 | ✅已完成 |
| 用户管理 | CRUD 操作 | 🔄开发中 |

## 3. 数据库设计
### users 表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |

## 4. API 接口
| 方法 | 路径 | 状态 |
|------|------|------|
| GET | /api/v1/users | ✅已完成 |
```

## 更新原则

- **准确**：每个关键节点都更新
- **具体**：不只是状态，还有具体内容
- **最小化**：只记录最核心的
