# 验证

## 概述

验证是确保代码正确实现的最后一道防线。

**核心原则：** 证据先于声明。

**铁律：**
```
没有新鲜验证证据，不得声称完成
```

---

## 【重要】输入输出文档规范

### 输入文档

**必须读取**： executing-plans 产出的文档

| 文档 | 位置 | 用途 |
|------|------|------|
| 计划文档 | `docs/plans/YYYY-MM-DD-<feature>.md` | 获取任务列表和验收标准 |
| 执行日志 | `docs/plans/YYYY-MM-DD-<feature>-execution-log.md` | 了解执行进度和遇到的问题 |

### 输出文档

**必须产出**：验收报告

| 文档 | 位置 | 用途 |
|------|------|------|
| 验收报告 | `docs/verification/YYYY-MM-DD-<feature>-report.md` | 记录验收结果、测试证据 |

### 文档内容模板

```markdown
# 验收报告：<功能名称>

> 创建时间：YYYY-MM-DD HH:MM

## 验收结果

| 验收项 | 状态 | 证据 |
|--------|------|------|
| 功能1 | ✅/❌ | 截图/日志 |
| 功能2 | ✅/❌ | 截图/日志 |

## 测试证据

### 接口检查
- [ ] 无 400/401/403/404/500 错误
- [ ] 控制台无 Error 日志

### 功能验证
- [ ] 功能1正常工作
- [ ] 功能2正常工作

### 构建验证
- [ ] 构建成功
- [ ] 测试通过

## 问题列表

| 问题 | 严重程度 | 状态 |
|------|----------|------|
| 问题描述 | 🔴高/🟡中/🟢低 | 待修复/已修复 |

## 结论

- [ ] 验收通过，可进入 code-review
- [ ] 验收不通过，需要修复
- [ ] 有条件通过（存在中/低优先级问题，需审查确认）
```

### 文档衔接

- **读取**：先读取 `docs/plans/` 下的计划文档和执行日志
- **产出**：验收报告保存到 `docs/verification/`
- **下一步**：code-review 会读取验收报告

## 验证流程

### 1. 验证前检查

**开始验证前，先确认以下条件：**

- [ ] executing-plans 已完成所有任务（计划中无 `[ ]` 标记）
- [ ] 执行日志已产出
- [ ] 代码已提交或暂存

### 2. 接口错误检查（必须）

**前端项目必须检查**：
1. 启动开发服务器：`npm run dev`
2. 打开浏览器控制台（F12）
3. 检查 Network 标签页
4. 确认所有 API 请求：
   - 无 400/401/403/404/500 等错误
   - 响应数据格式正确
5. 检查 Console 标签页
6. 确认无 Error 级别日志

### 3. 需求-实现对照

对照原始需求，验证每个功能点：

| 需求 | 实现状态 | 证据 |
|------|----------|------|
| 功能1 | ✅/❌ | 截图/测试结果 |
| 功能2 | ✅/❌ | 截图/测试结果 |

### 4. 运行验证命令

根据项目类型运行验证：

**前端项目：**
- 构建成功：`npm run build`
- E2E 测试：`npx playwright test`

**后端项目：**
- 测试通过：`pytest`
- API 测试：`curl` 验证

### 5. 验证失败处理流程

**如果验证失败，按以下流程处理：**

```
验证失败
    ↓
记录问题到"问题列表"
    ↓
判断问题严重程度
    ↓
├── 严重问题（阻塞）：
│   └── 返回 executing-plans 修复
│       └── 修复后重新验证（循环）
│
├── 中等问题（带条件通过）：
│   └── 记录到验收报告"待审查确认的问题"
│       └── 进入 code-review
│           └── 审查时决定是否修复
│
└── 轻微问题（可选修复）：
    └── 记录到验收报告
        └── 进入 code-review
            └── 可选择不修复
```

**严重程度判断标准：**

| 严重程度 | 标准 | 处理方式 |
|----------|------|----------|
| **高（阻塞）** | 功能完全不可用、核心流程断裂、数据错误 | 必须修复才能进入审查 |
| **中（条件）** | 非核心功能异常、体验问题、边界情况 | 可带条件进入审查 |
| **低（可选）** | 样式问题、文档缺失、非关键提示 | 可选择不修复 |

**重试机制：**
- 最大重试次数：3 次
- 每次重试后重新验证
- 超过 3 次失败，标记为"验证失败"并结束

## 验证清单

### 必须验证
- [ ] 功能正常运行
- [ ] 无接口错误
- [ ] 无控制台错误

### 建议验证
- [ ] 单元测试通过
- [ ] 构建成功
- [ ] 性能达标

### 严重程度标记

在验收报告中，对每个问题标记严重程度：

| 标记 | 含义 | 处理方式 |
|------|------|----------|
| 🔴 高 | 阻塞性问题 | 必须修复 |
| 🟡 中 | 非阻塞问题 | 可带条件通过 |
| 🟢 低 | 轻微问题 | 可选修复 |

---

## 有前端项目的特殊要求

> 注意：有前端的项目必须 E2E 测试

- 单元测试和组件测试无法发现运行时问题
- 代码看起来对，但实际跑起来可能有 bug
- 必须在真实浏览器环境中验证
