# 任务拆分

## 概述

将需求拆分为 2-5 分钟可完成的小任务。

**核心理念**：任务越小越容易执行和验证。

---

## 拆分原则

### 粒度标准

| 粒度 | 时间 | 说明 |
|------|------|------|
| 最佳 | 2-5 分钟 | 一个独立动作 |
| 可接受 | 5-10 分钟 | 多个小动作 |
| 太大 | >10 分钟 | 需要继续拆分 |

### 拆分维度

每个任务应该包含：
- **操作**：具体要做什么（创建/修改/删除/测试）
- **文件**：涉及的文件路径
- **验证**：完成后如何验证

---

## 拆分流程

1. **列出所有功能点**
   - 从需求文档提取所有功能
   - 不遗漏任何一个

2. **识别任务边界**
   - 每个任务 = 1 个独立操作
   - 可以单独验证

3. **细化任务步骤**
   - 操作 + 文件 + 验证

---

## 输出位置

**保存路径**：`docs/plans/<task>-plan.md`

---

## 输出格式

```markdown
### 任务 N：[任务名称]

**操作**：xxx
**文件**：xxx
**验证**：xxx

**步骤**：
1. xxx
2. xxx
```

---

## 示例

```
❌ 错误：任务太大
- [ ] 实现用户管理模块（30分钟）

✅ 正确：2-5分钟粒度
- [ ] 创建用户表结构
- [ ] 添加用户 Model
- [ ] 实现用户列表 API
- [ ] 实现用户创建 API
```

---

## 注意事项

- 不要把多个操作放在一个任务
- 每个任务必须可以单独验证
- 验证方式要明确
