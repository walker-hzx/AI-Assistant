# 意图分析

## 职责

分析用户需求，理解真实意图，判断任务类型和复杂度。

**定位**：管家的"输入处理"环节

---

## 输入处理

### 1. 表面 vs 真实意图

用户说的 ≠ 用户想要的

```
用户说："帮我优化下代码"
    ↓
真实意图可能是：
- 性能优化？
- 代码重构？
- 还是只是加个注释？
```

### 2. 追问技巧

如果意图不清晰，必须追问：

| 场景 | 应该问 |
|------|-------|
| 目标不明确 | "你希望通过这个达到什么效果？" |
| 范围不清晰 | "这个涉及哪些模块？" |
| 优先级不明 | "这个任务的紧急程度是？" |
| 成功标准不明 | "怎么算完成？" |

---

## 任务分类

### 四大类型

| 类型 | 特征 | 应该调度 |
|------|------|---------|
| **需求类** | 用户有想法但不确定 | 需求分析师 → brainstorming |
| **实现类** | 需求明确要实现功能 | 计划制定者 → writing-plans |
| **问题类** | 遇到错误/bug | 调试专家 → debugging |
| **优化类** | 改进现有代码/流程 | 分析具体目标再定 |

### 分类判断树

```
用户请求
    ↓
┌─────────────────┐
│ 是问题/错误吗？  │ → 是 → 问题类
└─────────────────┘
    ↓ 否
┌─────────────────┐
│ 有明确实现目标？ │ → 是 → 实现类
└─────────────────┘
    ↓ 否
┌─────────────────┐
│ 想改进/优化？    │ → 是 → 优化类
└─────────────────┘
    ↓ 否
需求类（需要进一步澄清）
```

## 复杂度评估

### 评估维度

| 维度 | 问题 |
|------|------|
| **范围** | 涉及多少模块/文件？ |
| **依赖** | 有多少外部依赖？ |
| **风险** | 修改可能影响什么？ |
| **验证** | 怎么验证完成？ |

### 复杂度分级

| 级别 | 特征 | 处理方式 |
|------|------|---------|
| **简单** | 单文件/单模块、可快速验证 | 直接处理，记录摘要 |
| **普通** | 多模块、有一定复杂性 | 调度 Skills，完整记录 |
| **复杂** | 跨系统、需要多阶段 | 调度 Skills，详细规划 |

### 复杂度判断

```
简单特征：
- [ ] 修改范围 ≤ 1 个文件
- [ ] 无数据库变更
- [ ] 无 API 变更
- [ ] 可在 5 分钟内完成

普通特征：
- [ ] 修改范围 2-5 个文件
- [ ] 可能有数据库/API 变更
- [ ] 需要测试验证

复杂特征：
- [ ] 修改范围 > 5 个文件
- [ ] 涉及数据库/架构变更
- [ ] 需要多阶段执行
```

## 代码探索（如需要）

### 何时使用 Explore

当用户描述涉及具体代码/功能时，先用 Explore 快速了解背景：

```
用户："帮我优化用户登录的性能"
    ↓
需要了解：用户登录相关代码在哪里？有多少文件？
    ↓
使用 Explore（quick）：
- 搜索用户登录相关代码
- 了解代码分布
    ↓
基于探索结果：
- 判断这是"优化类"任务
- 涉及 X 个文件
- 初步判断复杂度
```

### 代码探索指引

```
需要了解代码背景时：
- [ ] 用户提到了具体功能/模块
- [ ] 需要快速了解代码分布
- [ ] 评估任务复杂度

处理方式：
→ 直接分析（简单任务）
→ 调度 code-analyst（需要深入分析）
→ 记录关键信息供后续技能使用
```

### 输出格式

### 意图分析结果

```
## 意图分析

**用户表面需求**：[用户原话]

**真实意图**：[一句话说清楚要什么]

**代码背景**（如通过 Explore 了解）：
- 相关文件：[文件列表]
- 代码规模：[大/中/小]

**任务类型**：[需求类/实现类/问题类/优化类]

**复杂度**：[简单/普通/复杂]

**需要澄清的问题**：
- [ ] 问题1
- [ ] 问题2

**建议方案**：
- 调度：[角色/Skill]
- 记录级别：[摘要/完整/详细]
```
