# 意图分析

## 职责

分析用户需求，理解真实意图，判断任务类型和复杂度。

**定位**：管家的"输入处理"环节

---

## 输入处理

### 1. 表面 vs 真实意图

用户说的 ≠ 用户想要的

```
用户说："帮我优化下代码"
    ↓
真实意图可能是：
- 性能优化？
- 代码重构？
- 还是只是加个注释？
```

### 2. 追问技巧

如果意图不清晰，必须追问：

| 场景 | 应该问 |
|------|-------|
| 目标不明确 | "你希望通过这个达到什么效果？" |
| 范围不清晰 | "这个涉及哪些模块？" |
| 优先级不明 | "这个任务的紧急程度是？" |
| 成功标准不明 | "怎么算完成？" |

---

## 任务分类

### 四大类型

| 类型 | 特征 | 应该调度 |
|------|------|---------|
| **需求类** | 用户有想法但不确定 | 需求分析师 → brainstorming |
| **实现类** | 需求明确要实现功能 | 计划制定者 → writing-plans |
| **问题类** | 遇到错误/bug | 调试专家 → debugging |
| **优化类** | 改进现有代码/流程 | 分析具体目标再定 |

### 分类判断树

```
用户请求
    ↓
┌─────────────────┐
│ 是问题/错误吗？  │ → 是 → 问题类
└─────────────────┘
    ↓ 否
┌─────────────────┐
│ 有明确实现目标？ │ → 是 → 实现类
└─────────────────┘
    ↓ 否
┌─────────────────┐
│ 想改进/优化？    │ → 是 → 优化类
└─────────────────┘
    ↓ 否
需求类（需要进一步澄清）
```

## 【重要】角色选择边界

当需要调用分析类角色时，按以下边界选择：

### 边界定义

| 场景 | 应该调用 | 说明 |
|------|---------|------|
| 用户说"不知道怎么做"、"方向在哪" | thinking-coach | 思维层面困惑，需要厘清思路 |
| 用户说"有几个方案不知道选哪个" | strategist | 决策层面困惑，需要分析建议 |
| 用户说"想做个功能但不太清楚" | requirements-analyst | 需求层面模糊，需要具体化 |
| 需求已明确，需要拆解任务 | planner | 有需求了，需要变执行计划 |
| 有计划了，需要实现代码 | executor | 有计划了，需要执行 |

### 简单判断

```
用户问题：
- "不知道问题是什么" → thinking-coach（思维）
- "不知道选哪个方案" → strategist（决策）
- "需求不太清楚" → requirements-analyst（需求）
- "知道做什么，不知道怎么做" → planner（计划）
- "有计划了，帮我实现" → executor（执行）
```

## 任务处理模式

> **【重要】所有任务都必须走完整流程**

**取消快速模式**：无论任务大小，都必须走完整流程，确保：
- 意图分析明确任务目标
- 方案规划确定执行步骤
- 文档记录每个阶段产出
- 验证确保任务正确完成
- 审查确保代码质量

```
用户请求
    ↓
【管家】正在分析意图...
    ↓
【管家】意图分析完成
    ↓
【管家】下一步：交给 coordinator-planning 制定方案
```

### 完整模式处理

**所有任务走完整流程**：

```
【管家】收到需求：[用户原话]
    ↓
【管家】正在分析意图...
    ↓
【管家】意图分析完成
    ↓
【管家】下一步：交给 coordinator-planning 制定方案
```

> **注意**：即使是"简单"任务（如改变量名），也要创建文档记录，不能跳过流程

## 复杂度评估

### 评估维度

| 维度 | 问题 |
|------|------|
| **范围** | 涉及多少模块/文件？ |
| **依赖** | 有多少外部依赖？ |
| **风险** | 修改可能影响什么？ |
| **验证** | 怎么验证完成？ |

### 复杂度分级

| 级别 | 特征 | 处理方式 |
|------|------|---------|
| **简单** | 单文件/单模块、可快速验证 | 走完整流程（简化版） |
| **普通** | 多模块、有一定复杂性 | 走完整流程 |
| **复杂** | 跨系统、需要多阶段 | 走完整流程（详细版） |

> **所有级别都走完整流程**，区别只是文档详细程度和里程碑数量

### 复杂度判断

```
无论什么复杂度，都必须走完整流程：
- 简单任务：1-2 个里程碑（需求分析 → 执行）
- 普通任务：3-4 个里程碑（需求 → 计划 → 执行 → 验证）
- 复杂任务：5+ 个里程碑（需求 → 计划 → 执行 → 验证 → 审查）

普通特征：
- [ ] 修改范围 2-5 个文件
- [ ] 可能有数据库/API 变更
- [ ] 需要测试验证

复杂特征：
- [ ] 修改范围 > 5 个文件
- [ ] 涉及数据库/架构变更
- [ ] 需要多阶段执行
```

## 代码探索（如需要）

### 何时使用 Explore

当用户描述涉及具体代码/功能时，先用 Explore 快速了解背景：

```
用户："帮我优化用户登录的性能"
    ↓
需要了解：用户登录相关代码在哪里？有多少文件？
    ↓
使用 Explore（quick）：
- 搜索用户登录相关代码
- 了解代码分布
    ↓
基于探索结果：
- 判断这是"优化类"任务
- 涉及 X 个文件
- 初步判断复杂度
```

### 代码探索指引

```
需要了解代码背景时：
- [ ] 用户提到了具体功能/模块
- [ ] 需要快速了解代码分布
- [ ] 评估任务复杂度

处理方式：
→ 直接分析（简单任务）
→ 调度 code-analysis（需要深入分析）
→ 记录关键信息供后续技能使用
```
