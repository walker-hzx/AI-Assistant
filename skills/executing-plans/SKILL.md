# 执行计划

## 概述

加载计划，批判性审查，批量执行任务，批次之间报告以供审查。

**核心原则：** 批量执行，设置检查点供审查。**每完成一个任务都要进行学习型反思，不是机械地进入下一个任务。**

**开始时声明：** "我正在使用 executing-plans skill 来实现此计划。"

## 【重要】执行阶段职责

**此阶段职责**：具体实现 + 持续学习

| 职责 | 说明 |
|------|------|
| 边界条件实现 | 具体代码实现边界处理 |
| 异常处理实现 | 具体代码实现错误捕获 |
| 对照检查 | 完成后对照计划检查完整性 |
| 历史问题 | 检查类似问题是否重复 |
| 接口验证 | 确保接口无报错 |
| **学习型反思** | 每个任务完成后进行认知更新 |
| **假设验证** | 执行过程中验证计划中的假设 |
| **动态计划调整** | 基于执行发现主动调整后续步骤 |

**不在此阶段**：
- 需求确认 → brainstorming 阶段
- 任务拆分 → writing-plans 阶段

## 【重要】输入输出文档规范

### 输入文档

**必须读取**： writing-plans 产出的计划文档

| 文档 | 位置 | 用途 |
|------|------|------|
| 计划文档 | `docs/plans/YYYY-MM-DD-<feature>.md` | 获取任务列表和执行顺序 |
| 执行日志（可选） | `docs/execution/YYYY-MM-DD-*-execution-log.md` | 恢复之前的执行进度 |

### 输出文档

| 文档 | 位置 | 用途 |
|------|------|------|
| 执行日志 | `docs/execution/YYYY-MM-DD-<feature>-execution-N.md` | 记录执行进度、遇到的问题 |
| 计划更新 | `docs/plans/YYYY-MM-DD-<feature>.md` | 标记任务完成状态 `[x]` |

### 文档衔接

- **读取**：先读取 `docs/plans/` 下的计划文档
- **产出**：执行日志保存到 `docs/execution/`
- **下一步**：执行完成，进入验证阶段

## 【重要】执行日志

### 为什么需要执行日志

- 记录每个阶段做了什么，便于事后分析
- 追溯执行过程，发现问题环节
- 供 session-diagnosis/problem-analyzer 分析

### 执行日志模板

创建执行日志文件：`docs/execution/YYYY-MM-DD-<plan-name>-execution-log.md`

```markdown
# 执行日志: [计划名称]

> 创建时间：YYYY-MM-DD HH:MM

## 执行概览

| 项目 | 内容 |
|------|------|
| 计划文件 | docs/plans/YYYY-MM-DD-xxx.md |
| 开始时间 | YYYY-MM-DD HH:MM |
| 预计任务数 | N 个 |

## 阶段记录

### 阶段 1: [阶段名称]

**开始时间**: HH:MM
**操作**: [加载计划/执行任务/验证/里程碑评审]

#### 步骤记录

| 步骤 | 操作 | 调用工具 | 结果 |
|------|------|----------|------|
| 1.1 | 读取计划文件 | - | 成功 |
| 1.2 | 检查执行进度 | - | 从任务 N 开始 |
| 1.3 | 提取关键假设 | - | N 个假设待验证 |
| 1.4 | 执行任务 N | Task (子代理) | 完成 |

#### 阶段结果
- 状态：完成/暂停/问题
- 说明：[简述做了什么]

#### 学习反思
- 新认知：[这个阶段学到了什么]
- 假设验证：[哪些假设被验证/推翻]
- 计划影响：[是否触发了计划调整]
```

### 记录时机

| 时机 | 记录内容 |
|------|----------|
| 开始执行 | 加载计划、检查进度、预计任务数、关键假设清单 |
| 每个任务 | 开始、执行、调用工具、完成 |
| **任务完成后** | **学习反思结果：新认知、假设验证、计划影响** |
| 里程碑 | 暂停原因、评审结果、继续/调整、**认知更新总结** |
| 验证 | 运行什么验证、结果 |
| 完成 | 最终状态、发现的问题、**假设验证总结** |

### 粒度原则

```
✅ 记录：
- 阶段/步骤名称
- 开始时间、结束时间
- 调用了什么子代理/工具
- 阶段结果（成功/失败/问题）

❌ 不记录：
- 每个具体工具调用（Read/Edit）
- 生成的代码内容
- 思考过程细节
```

## 过程

### 步骤 1：加载并审查计划

1. 读取计划文件 (`docs/plans/YYYY-MM-DD-*.md`)
2. **提取关键假设**：从计划头部读取"关键假设"和"方案选型"，这些是执行过程中需要验证的
3. **检查执行进度**：查找 `[ ]` 和 `[x]` 标记，确认哪些任务已完成、哪些待执行
4. 从第一个未完成任务 `[ ]` 继续执行
5. **【重要】对照计划确认**：
   - 检查设计文档中的每个功能模块是否都有任务？
   - 是否有功能遗漏需要补充？
6. 批判性审查 - 识别关于计划的任何问题或顾虑
7. **【新增】审查方案合理性**：基于当前对代码的理解，计划选择的方案还合理吗？
8. 如果有顾虑：在实施前向你的人类伙伴提出
9. 如果没有顾虑：创建 TodoWrite 并继续
10. **【重要】创建执行日志**：创建 `docs/execution/YYYY-MM-DD-<plan-name>-execution-log.md`
11. **记录阶段 1**：记录加载计划、检查进度

### 步骤 2：执行批次

**默认：前 3 个任务**

对每个任务：
1. 标记为 in_progress
2. **【重要】任务执行前思考**：完成任务前思考清单
3. 完全按照每个步骤执行（计划中有小步骤）
4. 按指定运行验证
5. 标记为 completed
6. **同步更新计划文档**：将任务标记从 `[ ]` 改为 `[x]`
7. **【重要】记录到执行日志**：记录任务开始、执行、调用工具、完成

#### 【重要】任务执行前思考清单

**每个任务执行前，必须先完成以下思考：**

| 思考项 | 问题 | 我的回答 |
|--------|------|---------|
| **边界条件** | 输入的极端情况是什么？ | |
| **异常情况** | 可能出什么错？如何处理？ | |
| **依赖检查** | 依赖的模块/数据是否正常？ | |
| **测试策略** | 如何验证这个实现是正确的？ | |
| **历史问题** | 同类功能以前有什么问题？ | |
| **假设验证** | 这个任务依赖哪些假设？假设还成立吗？ | |

**思考要求**：
- 每个问题必须有明确回答（哪怕简单）
- 如果有不确定的，先搜索相关代码或问用户
- 思考结果记录到执行日志

**为什么需要思考**：
- 避免遗漏边界情况
- 提前发现潜在问题
- 确保测试覆盖

#### 【新增】任务完成后学习反思

**每个任务完成后，MUST 进行学习型反思（不是可选的）：**

```
【学习反思】任务 N 完成后

1. 新学到了什么？
   → 代码结构方面：...
   → 业务逻辑方面：...
   → 技术细节方面：...

2. 计划中的假设验证情况？
   → [已验证 ✓] 假设 X：确认成立
   → [已推翻 ✗] 假设 Y：实际情况是...
   → 影响评估：推翻的假设是否影响后续任务？

3. 对后续任务的影响？
   → 有没有任务变得更简单了？（可以合并/跳过）
   → 有没有任务变得更复杂了？（需要拆分/增加步骤）
   → 有没有发现新的必要任务？
   → 优先级是否需要调整？

4. 当前方案还是最优的吗？
   → 有没有发现更好的实现路径？
   → 需要切换方案吗？
```

**⚠️ 学习反思的结果 MUST 反映到计划调整中。如果发现需要调整，立即更新计划文档和 TodoWrite。**

**记录内容**：
- 任务名称
- 开始时间
- 执行前思考结果
- 调用了什么子代理/工具（如有）
- 完成状态
- 发现的问题（如有）

**同时更新进度追踪文件** `docs/execution/progress.md`：
- 记录任务开始时间
- 记录任务完成时间
- 更新任务状态
- 添加完成说明

### 步骤 3：报告

当批次完成时：
- 展示已实现的内容
- 展示验证输出
- 说："等待反馈。"
- **记录到执行日志**：记录批次完成状态

### 步骤 4：继续

根据反馈：
- 如果需要，应用更改
- 执行下一批次
- 重复直到完成

### 步骤 5：【重要】完成后验证

**所有任务完成后，必须调用验证**：

**【重要】验证前必须先检查以下内容**：

#### 1. 接口错误检查（前端项目）

在运行验证之前，先手动或通过 E2E 测试检查：
- 打开浏览器控制台（F12）
- 检查 Network 标签页
- 确认所有 API 请求无 4xx/5xx 错误
- 确认无控制台 Error 级别日志

#### 2. 历史遗留问题检查

对照原始需求，主动检查：
- 是否有历史遗留问题未修复
- 是否有同类问题在其他模块
- 是否有代码规范问题

#### 3. 然后调用正式验证

1. **verification** - 代码质量验证
   - 运行测试：`npm test` 或 `pytest`
   - Lint 检查：`npm run lint` 或 `ruff check`
   - 类型检查：`npm run type-check` 或 `mypy`
   - 构建成功：`npm run build`

2. **verification** - 需求验收
   - 对照需求文档检查功能
   - 确认 UI 正确显示
   - **【重要】逐条对照需求，检查是否有遗漏**

**验证流程**：
```
executing-plans 完成任务
    ↓
检查接口错误 + 历史遗留问题
    ↓
【重要】完成后告知用户：
"所有任务已完成，请调用 /verification 进行独立验证"
    ↓
用户调用 verification
    ↓
验证通过 → 进入 code-review
验证失败 → 返回执行阶段修复 → 再次验证
```

**【重要】执行阶段不内部调用 verification**：
- verification 是独立阶段，需要用户主动调用
- 执行阶段只负责完成代码实现
- 验证由独立的 verification 阶段负责

**重要**：
- ❌ 不能跳过 verification
- ❌ 不能说"完成"就结束
- ✅ 验证失败必须修复
- ✅ 全部通过才能算真正完成
- ✅ 完成后告知用户调用 verification

### 步骤 6：完成开发

所有任务完成并验证后：
- 调用 finishing-a-development-branch skill
- 遵循该技能验证测试、呈现选项、执行选择
- **【重要】最终更新执行日志**：记录完成状态、发现的问题、总结

### 步骤 7：同步进度（提醒）

执行完成后，提醒用户同步项目文档：

```
📌 建议执行：/sync-progress

完成了新功能，记得同步蓝图和需求状态，
避免后续会话读到过时的项目信息。
```

> 说明：`/sync-progress` 会把刚完成的工作同步到 `docs/蓝图.md` 和 `docs/requirements/README.md`，
> 保持项目知识准确。这一步不是强制的，但强烈建议在完成重要功能后执行。

## 混合执行模式

### 什么是混合执行

混合执行 = 根据任务依赖关系，智能调度并行/串行执行：
- **可并行任务**：无前置依赖、无共享资源冲突的任务，可以同时执行
- **串行任务**：有前置依赖或需要共享资源的任务，按顺序执行

### 何时使用混合执行

当计划中有多个独立任务时，混合执行可以显著提高效率：
- 前端组件开发（多个独立组件可并行）
- API 端点开发（多个独立端点可并行）
- 工具函数编写（多个独立函数可并行）

### 混合执行流程

**步骤 1：分析依赖**
- 读取计划中所有任务
- 分析每个任务的"依赖"字段：
  - 前置任务：必须先完成的任务
  - 可并行任务：可以同时执行的任务
  - 外部依赖：外部包、API、配置
- 构建任务依赖图

**步骤 2：识别可并行任务**
- 找出所有无前置依赖的任务
- 检查是否有共享资源冲突（如修改同一文件）
- 标记可并行执行的任务组

**步骤 3：执行并行任务**
- 同时启动多个子代理处理并行任务
- 每个子代理独立执行一个任务
- 等待所有并行任务完成

**步骤 4：执行串行任务**
- 按依赖顺序执行串行任务
- 如果某任务依赖的并行任务未完成，等待

**步骤 5：退化为串行**
- 如果没有可并行的任务（所有任务都有依赖）
- 自动切换为子代理驱动模式（串行执行）

### 并行度控制

- **最大并行数**：建议 2-4 个任务同时执行
- **过多并行的风险**：资源竞争、难以跟踪、合并冲突多
- **推荐策略**：同批次并行任务不超过 3 个

### 混合执行示例

```
任务 A（无依赖）→ 可并行
任务 B（无依赖）→ 可并行
任务 C（依赖 A）→ 串行
任务 D（依赖 A、B）→ 串行

执行顺序：
1. 并行执行 A、B
2. 等待 A、B 完成
3. 串行执行 C（依赖 A）
4. 串行执行 D（依赖 A、B）
```

## 里程碑评审点

### 什么是里程碑
里程碑 = 计划中的一组相关任务完成后的检查点。
例如：
- 完成一个功能模块
- 完成一个 API 端点
- 完成一组相关的代码修改

### 里程碑评审流程

每个里程碑完成后：
1. **暂停执行** - 不要立即开始下一个任务
2. **对照检查** - 对照原始需求检查当前进度
3. **评估状态** - 已完成 / 部分完成 / 需要调整
4. **与用户确认** - 确认后才能继续
5. **【重要】记录到执行日志**：记录里程碑评审原因、结果、用户反馈

### 里程碑评审检查清单

- [ ] 当前里程碑的所有任务完成了吗？
- [ ] 实现与原始需求一致吗？
- [ ] 有遗漏的功能吗？
- [ ] 需要调整后续计划吗？
- [ ] 用户确认可以继续吗？
- [ ] **【新增】假设验证总结**：里程碑涉及的假设，哪些被验证，哪些被推翻？
- [ ] **【新增】认知更新总结**：到目前为止最大的发现/学习是什么？对后续有什么影响？

### 里程碑评审 - 需求变更检查

在里程碑评审时，额外检查：

1. **扫描需求目录** - 检查 `docs/requirements/` 是否有新的相关需求
2. **检查需求变更** - 已有需求是否有修改
3. **识别冲突** - 新需求是否与当前实现冲突

如果发现需求变更：
- 记录变更内容
- 告知用户可能的冲突点
- 与用户确认是否需要调整计划

## 计划调整

### 何时需要调整

**主动调整（基于学习反思）**：
- 学习反思发现更优的实现路径
- 假设被推翻，影响后续任务
- 多个任务的学习结果揭示了新的模式或机会
- 认知深化后发现原计划有缺陷

**被动调整（意外事件）**：
- 发现计划遗漏的步骤
- 需求发生了变化
- 发现新的依赖或风险
- 实现过程中遇到未预见的难题

### 调整流程

1. **记录当前状态** - 暂停并记录已完成的工作
2. **分析差异** - 明确计划与实际的差异
3. **提出调整方案** - 列出需要添加/修改/删除的步骤
4. **与用户确认** - 调整方案需要用户批准
5. **更新计划** - 修改计划文档
6. **继续执行** - 按更新后的计划执行

### 调整原则

- 计划是活的，**应该随着认知深化而演化**
- 不要害怕暂停，停下来想清楚更重要
- 每次调整都要记录原因（学习反思结果 or 意外事件）
- 调整不是失败，是对信息更充分利用的表现
- 用户确认后才能继续

## 何时停止并寻求帮助

**立即停止执行当：**
- 批次中遇到阻碍（缺少依赖、测试失败、指令不清）
- 计划有严重缺陷阻止开始
- 你不理解某个指令
- 验证反复失败

**询问澄清而不是猜测。**

## 检查清单

执行前确认：
- [ ] 计划文件存在且可读
- [ ] 理解每个任务的步骤
- [ ] 知道如何验证每个任务

执行中确认：
- [ ] 按计划步骤执行
- [ ] 运行验证命令
- [ ] 记录任何问题

执行后确认：
- [ ] 所有测试通过
- [ ] 代码可运行
- [ ] 无引入新问题

## 工作流

```
writing-plans → executing-plans (里程碑评审 + 可调整)
                           ↓
              批次 1 → 报告 → 反馈
                           ↓
              批次 2 → 报告 → 反馈
                           ↓
              批次 N → 完成
```
