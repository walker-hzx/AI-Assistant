# 需求分析

## 概述

多角度分析需求，确保需求完整且可实现。

**核心理念**：从技术、业务、用户多个角度分析，补充需求中的遗漏。

---

## 分析维度

### 1. 技术角度

| 分析项 | 问题 |
|--------|------|
| 复杂度 | 技术实现复杂度 /5？ |
| 技术风险 | 有什么技术风险？ |
| 架构影响 | 对现有架构有什么影响？ |
| 依赖 | 依赖哪些现有模块/技术？ |

### 2. 业务角度

| 分析项 | 问题 |
|--------|------|
| 业务价值 | 解决什么业务问题？ |
| 优先级 | P0/P1/P2？ |
| 投入产出 | 投入产出比合理？ |
| 边界 | 什么情况做？什么不做？ |

### 3. 用户角度

| 分析项 | 问题 |
|--------|------|
| 目标用户 | 谁会用这个功能？ |
| 使用场景 | 在什么场景下使用？ |
| 用户体验 | 有什么体验要求？ |
| 反馈 | 用户有什么预期？ |

---

## 工作流程

1. **技术分析**
   - 评估实现复杂度
   - 识别技术风险
   - 确定架构影响

2. **业务分析**
   - 明确业务价值
   - 确认优先级
   - 划定边界

3. **用户分析**
   - 确定目标用户
   - 了解使用场景
   - 明确体验要求

## 适用场景

- 需求理解阶段完成
- 需要从多角度审视需求
- 需求较复杂需要全面分析

---

## 注意事项

- 不要只从技术角度分析
- 业务和用户角度同样重要
- 分析要有依据，不要猜测
