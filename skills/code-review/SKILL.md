# 代码审查

## 概述

代码审查是确保代码质量的重要环节。

**核心原则：** 尽早审查，频繁审查。

---

## 【重要】输入输出文档规范

### 输入文档

**必须读取**： verification 产出的验收报告

| 文档 | 位置 | 用途 |
|------|------|------|
| 验收报告 | `docs/verification/YYYY-MM-DD-<feature>-verification.md` | 获取验收结果和测试证据 |
| 计划文档 | `docs/plans/YYYY-MM-DD-<feature>.md` | 了解原始任务列表 |

### 输出文档

**必须产出**：审查报告

| 文档 | 位置 | 用途 |
|------|------|------|
| 审查报告 | `docs/reviews/YYYY-MM-DD-<feature>-review.md` | 记录审查结果、问题、学到的经验 |

### 文档内容模板


### 文档衔接

- **读取**：先读取 `docs/verification/` 下的验收报告
- **产出**：审查报告保存到 `docs/reviews/`
- **结束**：整个开发流程完成

## 审查流程

### 1. 自我审查（提交前）

在调用 reviewer agent 之前，先进行自我审查：

**常见问题检查：**
- [ ] 是否有明显的语法错误
- [ ] 是否有 console.log 或调试代码
- [ ] 是否有未使用的变量/导入
- [ ] 是否有 TODO 待完成

**本地验证：**
- [ ] 测试是否通过
- [ ] Lint 检查是否通过
- [ ] 类型检查是否通过
- [ ] 构建是否成功

**筛选重点：**
- 如果自检发现很多问题 → 先修复，再请求审查
- 如果自检基本没问题 → 可以请求审查

### 2. 提交审查

**获取 git 信息：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)
HEAD_SHA=$(git rev-parse HEAD)
```

**调用审查：**
```bash
Task: reviewer
描述: 审查刚完成的xxx功能
```

### 3. 处理反馈

收到审查反馈后：
1. 理解每个问题
2. 区分必须修复 vs 建议修复
3. 修复后重新审查（或确认修复）

---

## 审查内容

### 必须检查
- [ ] 功能正确性
- [ ] 代码安全性
- [ ] 错误处理
- [ ] 测试覆盖

### 建议检查
- [ ] 代码可读性
- [ ] 命名规范
- [ ] 性能考虑
- [ ] 文档更新

---

## 何时审查

**必须审查：**
- 完成主要功能后
- 合并到 main 分支前
- 提交 PR 前

**可选但有价值：**
- 遇到困难时（获得新视角）
- 重构前（基线检查）
- 修复复杂 bug 后
