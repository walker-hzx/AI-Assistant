# 需求-实现对照验收

## 概述

在代码完成后，对照原始需求验证实现完整性。

**核心原则：** 需求即契约，实现必须满足契约。

**开始时声明：** "我正在使用 execution-validation skill 进行需求-实现对照验收。"

## 【重要】执行前自检（必须步骤）

> **注意**：以下自检是**必须步骤**，不能跳过！

在开始验收之前，必须先完成以下自检：

### 1. 【必须】接口错误检查

**这是必须步骤，不能跳过！**

**前端项目必须检查**：
1. 启动开发服务器：`npm run dev`
2. 打开浏览器控制台（F12）
3. 检查 Network 标签页
4. 确认所有 API 请求：
   - 无 400/401/403/404/500 等错误
   - 响应数据格式正确
5. 检查 Console 标签页
6. 确认无 Error 级别日志

**如果有接口错误**：
- 必须先修复
- 修复后重新检查
- 确认无错误后才能继续

**输出格式**：
```markdown
### 接口错误检查结果

| API 端点 | 状态码 | 响应时间 | 结果 |
|----------|--------|----------|------|
| /api/users | 200 | 120ms | ✅ |
| /api/login | 200 | 80ms | ✅ |
| /api/data | 500 | 500ms | ❌ |

**发现的问题**：/api/data 返回 500 错误 → 已修复，重新检查通过
```

### 2. 历史遗留问题扫描

对照原始需求，主动扫描：
- 是否有历史遗留问题未修复
- 是否有同类问题在其他模块（主动搜索，不只是检查当前修改的代码）
- 是否有代码规范问题

**扫描方式**：
1. 搜索相关模块的旧代码
2. 检查是否有类似的 bug 模式
3. 检查是否有配置遗漏

**输出格式**：
```markdown
### 历史遗留问题检查

| 问题 | 位置 | 状态 |
|------|------|------|
| 同类 bug | src/utils/helper.ts | ❌ 未修复 |
| 配置遗漏 | src/config/app.ts | ✅ 已修复 |
```

### 3. 需求完整性自检

在对照需求文档之前，先自检：
- 是否有需求理解偏差
- 是否有功能遗漏
- 是否有边界条件未处理

**输出格式**：
```markdown
### 自检结果

- 接口错误：N 个
- 历史问题：N 个
- 需求遗漏：N 个
```

**重要**：
- ❌ 不能跳过自检直接开始验收
- ✅ 自检发现问题先修复
- ✅ 自检通过后再正式验收

## 为什么需要这个环节

- 计划可能遗漏细节
- 实现过程中可能产生偏差
- 验证代码质量 ≠ 验证需求满足
- 确保交付物与预期一致

## 工作流程

### 0. 【重要】执行前自检

> 必须先完成自检，才能开始正式验收

按照上方"执行前自检"章节完成：
1. 接口错误检查
2. 历史遗留问题扫描
3. 需求完整性自检

### 1. 读取原始需求

查找并读取：
- Design doc: `docs/plans/*-design.md`
- 或 brainstorming 记录
- 提取验收标准
- 提取功能范围（模块、功能点）

如果找不到原始需求：
- 询问用户原始需求是什么
- 或根据已有代码推断需求

### 2. 【重要】逐项对照检查

对每个需求项检查：
- 是否有对应的代码实现
- 实现是否完整
- 是否有遗漏的功能

**【重要】主动检查遗漏**：
- 是否有 P0 功能未实现
- 是否有边界条件未处理
- 是否有错误处理缺失
- 是否有同类问题在其他地方（主动搜索）

**【重要】主动检查历史问题**：
- 这个问题是否在旧代码中有同类问题
- 修改是否引入了新的同类问题
- 是否有配置/依赖遗漏

输出对照表：
| 需求项 | 实现状态 | 说明 |
|--------|----------|------|
| 功能 A | ✅ 已实现 | 位置：xxx |
| 功能 B | ⚠️ 部分实现 | xxx |
| 功能 C | ❌ 未实现 | 原因：xxx |

### 3. 【重要】逐项对照检查（必须有证据）

对每个需求项检查：
- 是否有对应的代码实现
- 实现是否完整
- 是否有遗漏的功能

**【重要】每项需求必须提供实现证据**：

| 需求项 | 实现状态 | 代码位置（证据） | 说明 |
|--------|----------|------------------|------|
| 功能 A | ✅ 已实现 | `src/components/Button.tsx:45-67` | 实现了点击事件 |
| 功能 B | ⚠️ 部分实现 | `src/api/user.ts:23-45` | 缺少权限验证 |
| 功能 C | ❌ 未实现 | - | 原因：xxx |

**证据要求**：
- 必须提供具体的文件路径和行号
- 如果是 UI 功能，必须提供截图或录屏
- 如果是 API 功能，必须提供接口响应示例
- 没有证据的需求项视为未完成

**【重要】主动检查遗漏**：
- 是否有 P0 功能未实现
- 是否有边界条件未处理
- 是否有错误处理缺失
- 是否有同类问题在其他地方（主动搜索）

**【重要】主动检查历史问题**：
- 这个问题是否在旧代码中有同类问题
- 修改是否引入了新的同类问题
- 是否有配置/依赖遗漏

### 4. 【重要】输出验收报告

```markdown
## 验收报告

### 自检结果
- 接口错误：N 个（已修复/未修复）
- 历史问题：N 个（已修复/未修复）
- 需求遗漏：N 个

### 需求对照表（每项必须有证据）

| 需求项 | 状态 | 代码位置 | 截图/证据 |
|--------|------|----------|-----------|
| 需求 1：xxx | ✅ 已实现 | `src/...:xx-xx` | [截图] |
| 需求 2：xxx | ⚠️ 部分实现 | `src/...:xx-xx` | - |
| 需求 3：xxx | ❌ 未实现 | - | - |

### 功能演示（有前端项目必须）

**环境**：浏览器 Chrome / URL: http://localhost:5173

| 功能 | 截图 | 说明 |
|------|------|------|
| 登录页面 | [截图] | 正确渲染 |
| 登录成功 | [截图] | 跳转正确 |

### 历史遗留问题
- [ ] 问题 1：位置 xxx (状态)

### 结论
[是否可以验收 / 需要补充什么]
```

**重要**：
- ❌ 没有代码位置的需求项不算完成
- ❌ 没有截图的前端功能不算完成
- ✅ 用户确认后才能标记为完成

## 验收标准

- 每个原始需求都有明确状态
- 未完成项有合理解释
- 用户确认后才能标记为完成

## 常见问题

### Q: 找不到原始需求文档
A: 询问用户，或根据代码实现反推需求

### Q: 实现与需求有偏差
A: 记录偏差项，与用户确认是否接受

### Q: 需求有遗漏
A: 标记为未完成，说明需要补充

## UI 实际运行验证（有前端项目）

> 注意：此环节仅适用于有前端的项目。纯后端项目跳过 UI 实际运行验证，直接进行需求对照。

### 什么是 UI 实际运行验证

在真实浏览器环境中运行应用，检查：
- 页面是否正确渲染
- 交互是否正常工作
- 数据是否正确显示
- 错误是否正确处理

### 验证方式

1. **启动开发服务器**
```bash
npm run dev
```

2. **手动验证或 E2E 测试验证**
- 运行关键流程的 E2E 测试
- 检查测试截图
- 验证 UI 元素正确渲染

3. **截图对比**（可选）
- 预期效果截图
- 实际效果截图
- 对比差异

### 检查要点

| 检查项 | 说明 |
|--------|------|
| 页面加载 | 无白屏、无控制台错误 |
| 组件渲染 | 正确显示、无样式问题 |
| 交互响应 | 点击/输入有正确响应 |
| 数据展示 | 数据显示正确、无截断 |
| 错误处理 | 错误提示正确显示 |

### 输出格式

```markdown
## UI 实际运行验证结果

### 验证环境
- 浏览器：Chrome
- 分辨率：1920x1080
- URL：http://localhost:5173

### 验证结果
| 功能 | 状态 | 说明 |
|------|------|------|
| 登录页面 | ✅ | 正确渲染，输入框可交互 |
| 登录成功 | ✅ | 提交后正确跳转 |
| 登录失败 | ✅ | 错误提示正确显示 |
| 用户头像 | ✅ | 正确显示用户名称 |

### 结论
[是否通过验收]
```

## 验收标准

- 每个原始需求都有明确状态
- 未完成项有合理解释
- 用户确认后才能标记为完成
- UI 实际运行验证通过（有前端项目则必须）
- **接口错误检查通过**
- **历史遗留问题已扫描并记录**

## 【重要】主动检查意识

### 为什么需要主动检查

- 计划可能遗漏细节
- 实现过程中可能产生偏差
- 历史遗留问题可能影响新功能
- 接口错误可能不容易被发现

### 主动检查清单

- [ ] 是否有同类 bug 在其他模块（主动搜索）
- [ ] 是否有配置/依赖遗漏
- [ ] 是否有边界条件未处理
- [ ] 是否有错误处理缺失
- [ ] 是否有 API 调用报错（4xx/5xx）
- [ ] 是否有控制台 Error 日志

### 检查原则

- **不只是检查当前修改的代码**
- **要搜索相关模块的旧代码**
- **要主动问"还有没有类似问题"**

## 与其他环节的关系

- **之前**：verification-before-completion（代码质量验证）
- **之后**：update-blueprint（更新蓝图）
- **区别**：质量验证 vs 需求验证

## 关键原则

1. **逐项检查**：每个需求都要有明确状态
2. **证据说话**：用代码位置、实现细节作为证据
3. **诚实报告**：未完成就是未完成，不掩盖
4. **用户确认**：最终由用户决定是否可以验收
