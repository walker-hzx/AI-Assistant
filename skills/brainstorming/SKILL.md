# 头脑风暴：快速需求分析

## 概述

快速需求分析，核心是"3个问题"。

**核心理念**：
- 快速澄清，不追求完美文档
- 按需生成文档，不要为了文档而文档
- AI Agent 开发不需要传统开发那样复杂的规范

---

## 快速流程（3个核心问题）

### 问题 1：做什么
一句话说明白要做什么。

### 问题 2：为什么做
不做会怎样？解决什么问题？

### 问题 3：怎么做
最小实现方案是什么？

## 文档策略

### 【重要】必须产出需求文档

**原因**：确保流程信息完整传递，供后续阶段读取。

| 情况 | 文档 |
|------|------|
| 简单需求 | `docs/intent/YYYY-MM-DD-<feature>.md`（1-5行） |
| 中等复杂度 | `docs/intent/YYYY-MM-DD-<feature>.md`（10-20行） |
| 复杂需求 | `docs/intent/YYYY-MM-DD-<feature>.md`（核心要点） |

### 文档位置与命名

```
docs/intent/
└── YYYY-MM-DD-<feature>.md
```

**示例**：`docs/intent/2026-02-25-user-auth.md`

### 文档内容模板

```markdown
# 需求：<功能名称>

## 1. 做什么
[一句话说明白要做什么]

## 2. 为什么做
[不做会怎样？解决什么问题？]

## 3. 验收标准
- [ ] 标准1
- [ ] 标准2
- [ ] 标准3

## 4. 最小实现方案
[最简实现思路]

## 5. 多角度分析（可选）
如果需求较复杂，补充以下分析：

### 技术角度
- 复杂度：/5
- 技术风险：
- 架构影响：

### 业务角度
- 业务价值：
- 优先级：P0/P1/P2
- 投入产出：

### 用户角度
- 目标用户：
- 使用场景：

### 风险角度
- 假设：
- 遗漏：
- 风险：

## 6. Team 建议（可选）
- [ ] 建议升级到 requirements-incubation
- [ ] 建议升级到 requirements-review
- [ ] 当前复杂度可继续

## 7. 风险/注意事项
[可选：技术风险、依赖等]
```

### 不需要的文档

- ~~详细设计文档~~（除非用户明确要求）
- ~~蓝图规范~~
- ~~规约文档~~

```

```

## 【增强】多角度思考清单

**在分析需求时，自动从以下角度思考**：

### 1. 技术角度
- [ ] 实现复杂度高吗？（1-5分）
- [ ] 有什么技术风险？
- [ ] 对现有架构有什么影响？
- [ ] 有什么外部依赖？
- [ ] 需要多长时间/工作量？

### 2. 业务角度
- [ ] 业务价值是什么？
- [ ] 目标用户是谁？
- [ ] 优先级如何判定？（P0/P1/P2）
- [ ] 投入产出比如何？
- [ ] 不做有什么后果？

### 3. 用户角度（含用户流程检查）
- [ ] 用户场景是否清晰？
- [ ] 有什么可用性问题？
- [ ] 用户痛点是否被解决？
- [ ] 有什么体验隐患？

**【新增】用户流程检查清单**（涉及前端功能时必须检查）：
- [ ] **用户操作路径**：用户从哪进？做什么？然后呢？（完整路径）
- [ ] **核心操作**：最常用的3个操作是什么？
- [ ] **状态反馈**：操作后用户能看到什么反馈？
- [ ] **异常流程**：操作失败/出错时怎么处理？
- [ ] **边界提示**：有什么限制需要告知用户？

> **注意**：只检查"用户流程"概念层，具体界面设计在 writing-plans 阶段

### 4. 风险角度
- [ ] 需求假设是什么？这些假设成立吗？
- [ ] 有什么遗漏的场景/用户/情况？
- [ ] 需求内部是否矛盾？
- [ ] 失败了有什么后果？

## 【增强】场景识别：是否需要升级到 Team

**在分析过程中，自动判断是否建议使用 Team**：

### 建议升级到 Team 的情况

| 指标 | 阈值 | 建议 |
|------|------|------|
| 涉及模块数 | > 3个 | 建议 Team |
| 技术复杂度 | 4-5分 | 建议 Team |
| 架构变更 | 是 | 必须 Team |
| 不确定地方 | > 3个 | 建议 Team |
| 业务价值 | 高 | 建议 Team |
| 预算时间 | > 1天 | 建议 Team |

### 判断流程

```
需求分析
    ↓
评估复杂度
    ↓
├── 简单（< 2个指标触发）→ 继续 brainstorming
│
└── 复杂（≥ 2个指标触发）→ 建议用户：
    "这个需求比较复杂，建议使用 Team 进行多角度分析：
    - requirements-incubation（需求孵化）
    - requirements-review（需求评审）"
```

### Team 选择建议

| 场景 | 推荐 Team |
|------|-----------|
| 想法不清晰，需要多角度分析 | requirements-incubation |
| 需求较明确，需要评审验证 | requirements-review |
| 非常重要，需要孵化+评审 | 两者都用 |

## 增强后的工作流程

```
1. 快速了解背景
2. 问3个核心问题（做什么/为什么/怎么做）
3. 【自动】多角度思考清单检查
4. 【自动】场景复杂度评估
5. 方案对比（可选）
6. 确认设计
7. 产出文档

【如果复杂 → 建议升级 Team】
【如果简单 → 直接进入计划阶段】
```
