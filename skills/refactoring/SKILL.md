# 重构

> 安全、渐进地改善代码结构，不改变外部行为。

---

## 重构前检查

**必须全部确认后才能开始重构：**

- [ ] **测试覆盖**：要重构的代码有测试吗？没有 → 先补测试再重构
- [ ] **影响范围**：涉及几个文件？几个调用方？范围超过 5 个文件 → 分批处理
- [ ] **重构目标**：为什么要重构？达到什么效果？（不能只说"代码不好看"）
- [ ] **当前状态**：所有测试通过（绿灯起步）

---

## 重构流程

```
绿灯 → 单步重构 → 运行测试 → 绿灯 → 下一步
                       ↓
                  测试失败 → 立即回退这一步
```

### 具体步骤

1. **确认绿灯** — 运行所有相关测试，确保全部通过
2. **单步重构** — 一次只做一种重构操作（不混合多种）
3. **立即验证** — 每步后运行测试
4. **测试失败** → 回退这一步，分析为什么失败，调整策略
5. **测试通过** → 继续下一步重构
6. **完成后全量验证** — 运行完整测试套件

## 常见重构操作

### 提取类

| 操作 | 适用场景 | 注意 |
|------|---------|------|
| 提取函数/方法 | 函数过长、重复代码 | 命名要体现意图 |
| 提取组件（前端） | 组件过大、可复用的 UI 片段 | Props 接口要清晰 |
| 提取常量/配置 | 硬编码的值 | 放在合适的作用域 |
| 提取类型/接口 | 重复的类型定义 | 放在 types/ 目录 |

### 简化类

| 操作 | 适用场景 | 注意 |
|------|---------|------|
| 内联临时变量 | 变量只用一次且不增加可读性 | 确认内联后仍可读 |
| 简化条件（guard clause） | 深层嵌套的 if/else | 提前返回减少嵌套 |
| 替换条件为多态/策略 | 大量 switch/if-else 分支 | 确认分支会继续增长 |
| 消除重复代码（DRY） | 多处相同逻辑 | 只合并真正相同的，不要过度抽象 |

### 移动类

| 操作 | 适用场景 | 注意 |
|------|---------|------|
| 移动函数到更合适的模块 | 函数和所在模块关系不大 | 更新所有 import |
| 重命名（变量/函数/文件） | 命名不体现意图 | 全局搜索确保全部更新 |
| 拆分文件 | 文件 > 400 行 | 按职责拆，不是按行数 |

## 重构范围控制

### 必须遵守

- **单次只处理一种重构** — 不要同时提取函数又改名又移动文件
- **不在重构中混入功能改动** — 重构是改结构，不是改行为
- **大范围重构分多次** — 每次一个可独立验证的单元

### 范围评估

| 影响范围 | 策略 |
|---------|------|
| 1-2 个文件 | 一次完成 |
| 3-5 个文件 | 分 2 批，每批验证 |
| 6+ 个文件 | 制定重构计划，按模块分批 |

---

## 前端重构要点（Vue 3）

- 组件拆分时保持 Props 接口稳定
- 提取 composable 时注意响应式依赖
- 样式提取用 Tailwind 类而非自定义 CSS
- 注意 provide/inject 的依赖链

## 后端重构要点（Python FastAPI）

- 拆分 service 时保持接口签名不变
- 移动模型时更新 Alembic migration
- 提取工具函数到 utils/
- 注意依赖注入的影响

---

## 输出标准

重构完成后汇报：
- 重构了什么（目标和动机）
- 改动了哪些文件
- 测试验证结果
- 重构前后对比（结构更清晰了哪里）
- 意外发现（如果有）
