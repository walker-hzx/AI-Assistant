# 进度同步

> 将最近完成的工作同步到蓝图，更新需求状态，让项目知识始终准确。

---

## 使用时机

- 完成一个功能后（`/executing-plans` 结束后）
- 需求状态发生变化后（新增、作废、调整）
- 做了重要的技术决策后
- 隔一段时间后，蓝图可能已过时

---

## 工作流程

### Step 1：读取当前状态

并行读取以下文件，了解当前真实状态：

1. **蓝图** — `docs/蓝图.md`：当前记录的项目状态
2. **需求总览** — `docs/requirements/README.md`：当前需求列表和状态
3. **最近执行日志** — `docs/execution/` 目录下最新的文件：最近完成了什么
4. **最近计划文档** — `docs/plans/` 目录下最新的文件：哪些任务已完成（标记了 `[x]`）

### Step 2：识别差异

对比"执行日志/计划完成情况"与"蓝图当前记录"，找出需要更新的内容：

| 检查项 | 说明 |
|--------|------|
| **新完成的功能** | 执行日志里完成了，但蓝图功能清单里还是"开发中"或"待开发" |
| **新增的 API** | 实现了新接口，蓝图 API 表还没有 |
| **需求状态变化** | 某需求已实现，requirements/README.md 还没标记完成 |
| **已作废的内容** | 需求变了，蓝图里有过时的功能或 API |
| **新的技术决策** | 选了某个方案，蓝图里没有记录 |
| **新增的约定** | 开发过程中确立了某个约定，没有记录 |

### Step 3：生成更新草稿

**不要直接修改文件。先生成草稿展示给用户确认。**

以如下格式展示需要更新的内容：

```
## 同步草稿

### 蓝图更新

**功能清单变更**：
- `用户登录` 状态：🔄开发中 → ✅已完成
- 新增 `消息推送` 功能（代码位置：src/services/notification.ts）

**新增 API**：
| 方法 | 路径 | 说明 | 状态 |
|------|------|------|------|
| POST | /api/v1/notifications | 发送通知 | ✅已完成 |

**关键决策记录**（新增）：
- 选择 WebSocket 而非轮询实现实时通知（原因：性能需求）

### 需求状态变更

- `消息推送需求` 状态：🔄进行中 → ✅已完成

---
确认后更新以上内容？(y/n)
```

### Step 4：执行更新

用户确认后，更新以下文件：

1. **`docs/蓝图.md`** — 更新功能清单状态、API 接口、关键决策等
2. **`docs/requirements/README.md`** — 更新需求状态

**更新原则**：
- 增量更新，不重写整个文件
- 只更新有变化的部分
- 保留原有的结构和内容
- 在蓝图末尾的"更新记录"表格追加本次同步记录

### Step 5：确认

展示更新后的关键段落，确认写入正确。

---

## 输出格式

同步完成后，输出简洁的同步报告：

```
✅ 进度同步完成

同步内容：
- 蓝图功能清单：2 项更新
- API 接口：3 条新增
- 需求状态：1 项变更

下次同步建议：完成下一个功能模块后
```

---

## 注意事项

- **不强制重写蓝图** — 只更新有变化的部分
- **保留历史决策** — 关键决策记录不删除，只追加
- **需求状态以 requirements 为准** — 如果不确定某需求状态，以 `docs/requirements/README.md` 为准
- **轻量原则** — 同步应该快速、低摩擦，不要变成一个复杂的流程
