# 方案制定

## 职责

基于意图分析结果，制定执行方案，选择角色组合，安排执行顺序。

**定位**：管家的"规划"环节

---

## 【重要】输入输出

### 输入
接收 coordinator-intent 的分析结果：
- 真实意图
- 任务类型
- 复杂度
- 处理模式

### 输出
制定执行方案，决定派发给谁

---

## 角色选择

### 角色 → Skill 映射

| 任务类型 | 需要的能力 | 调度的 Skill/Agent |
|----------|-----------|-------------------|
| 需求类 | 需求分析 | brainstorming |
| 需求类 | 需求确认 | requirement-validation |
| 实现类 | 制定计划 | writing-plans |
| 实现类 | 执行计划 | executing-plans |
| 问题类 | 定位问题 | debugging |
| 问题类 | 分析代码 | code-analysis |
| 优化类 | 性能优化 | performance-expert |
| 优化类 | 重构 | refactoring-expert |
| 验证类 | 功能验证 | verification |
| 验证类 | 安全审查 | security-review |
| 审查类 | 代码审查 | code-review |
| 通用 | 思维困惑 | thinking-coach |

### 按需专业角色

| 场景 | 触发条件 | 调用的专家 |
|------|---------|-----------|
| 技术选型/架构设计 | 新系统、技术架构重大变更 | architect |
| 数据库设计/优化 | schema 变更、慢查询、数据迁移 | database-expert |
| 性能问题 | 有性能指标要求、需要性能优化 | performance-expert |
| 代码重构 | >5个文件、跨模块重构 | refactoring-expert |
| 安全审查 | 用户输入、认证、支付、敏感数据 | security-reviewer |
| 测试设计 | 复杂业务流程、需要 E2E 测试 | test-designer |
| 代码分析 | 前后端接口分析、问题诊断 | code-analysis |
| 项目调研 | 中途接手新项目、需要了解现状 | project-researcher |
| 网页研究 | 需要从网页获取信息、分析文档 | web-researcher |
| E2E 测试 | 需要验证用户完整流程、浏览器测试 | e2e-tester |

**触发判断流程**：
```
收到任务
    ↓
检查是否涉及数据库 schema？ → 是 → 调用 database-expert
    ↓
检查是否有性能要求？ → 是 → 调用 performance-expert
    ↓
检查是否涉及安全敏感操作？ → 是 → 调用 security-reviewer
    ↓
检查是否大规模重构？ → 是 → 调用 refactoring-expert
    ↓
继续常规流程
```

**注意**：专业角色不是替代常驻角色，而是增强：
```
主流程：需求 → 计划 → 执行 → 验证
            ↓           ↓           ↓
         [architect]  [database-expert]  [performance-expert]
         [技术选型]   [schema设计]       [性能优化]
            ↓           ↓           ↓
继续计划 → 继续执行 → 继续验证
```

## 方案结构

### 阶段划分

```
阶段 1：需求/分析（如果是需求类/问题类）
    ↓
阶段 2：规划（如果是实现类）
    ↓
阶段 3：执行
    ↓
阶段 4：验证
    ↓
阶段 5：审查（可选）
```

### 里程碑设计

每个阶段是一个里程碑：

```
## 里程碑 1：[名称]

- **目标**：[阶段目标]
- **调度**：[使用的 Skill/Agent]
- **产出**：[预期交付物]
- **验证**：[如何验证完成]
```

---

## 执行顺序

### 常见模式

| 模式 | 适用 | 顺序 |
|------|------|------|
| 需求→计划→执行→验证 | 新功能开发 | 线性 |
| 分析→修复→验证 | Bug 修复 | 线性 |
| 分析→优化→验证 | 性能优化 | 线性 |
| 需求→审查→计划→执行→验证 | 重大功能 | 线性+审查 |

### 并行判断

```
两个任务可以并行当：
- [ ] 无依赖关系
- [ ] 不修改同一文件
- [ ] 独立验证

示例：
- [ ] 写文档 + [ ] 准备测试数据 → 可并行
- [ ] 修改 API + [ ] 修改前端调用 → 串行
```

## 规划前研究（如需要）

### 何时使用 Plan

当需要制定详细实施计划时，先用 Plan 系统性了解项目：

```
任务：实现用户权限管理功能
    ↓
需要了解：
- 现有权限系统怎么实现的？
- 有哪些模块？依赖关系？
- 哪些文件可能会受影响？
    ↓
使用 Plan：
- 研究项目中的权限相关代码
- 分析依赖关系
- 识别潜在风险点
    ↓
基于研究结果：
- 制定包含权限模块改造的详细计划
- 识别需要协同修改的文件
- 预判可能的依赖问题
```

### 规划研究指引

```
需要系统性了解项目时：
- [ ] 任务涉及多个模块
- [ ] 需要了解依赖关系
- [ ] 需要分析项目结构
- [ ] 任务是"实现类"且复杂度较高

处理方式：
→ 使用本技能（coordinator-planning）进行研究
→ 必要时调度 code-analysis 协助分析
→ 输出完整的方案文档
```
