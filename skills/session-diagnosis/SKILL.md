# 协作复盘诊断

## 概述

在项目或大功能完成后，回顾整个协作过程，诊断问题类型并给出优化建议。

**触发时机**：
- 项目完成时
- 大功能完成时
- 你说"开始记录"时

---

## 【重要】持续记录机制

### 触发方式

当你说"开始记录"时，触发记录机制

### 存放位置

项目/.claude/learnings/session-notes.md

例如：你的实践项目/agent-monorepo/.claude/learnings/session-notes.md

### 记录格式

```markdown
# 对话记录

## 会话 YYYY-MM-DD

### 对话 1
**你**: [问题/反馈]
**我**: [回答]

### 对话 2
**你**: [问题/反馈]
**我**: [回答]

...
```

### 工作流程

1. 你说"开始记录"
2. 我创建 `.claude/learnings/session-notes.md`
3. 每次对话后自动追加到文件
4. 复盘时调用 session-diagnosis 分析

### 复盘时

调用 session-diagnosis：
- 读取 session-notes.md
- 一起分析历史记录
- 给出改进建议
- 大功能完成时
- 用户主动调用时说"诊断一下"

## 诊断维度

| 类型 | 说明 |
|------|------|
| **配置问题** | 插件 skill 能力不足，无法覆盖某些场景 |
| **使用问题** | 用户使用 skill 的方式可以更高效 |
| **协作问题** | 用户提问/描述问题的方式可以改进 |
| **代码问题** | 项目本身的实现问题（不是配置问题） |

## 工作流程

### Step 1: 了解背景

询问用户（一次一个）：

1. **做什么**：这个项目/功能是做什么的？
2. **用了什么 skill**：过程中用了哪些 skill？（如 debugging, brainstorming 等）
3. **遇到什么问题**：过程中遇到的最大问题是什么？
4. **怎么解决的**：问题是怎么解决的？

### Step 2: 分析问题

根据用户描述，分析每个问题属于哪个维度：

**配置问题特征**：
- 某个场景 skill 没有覆盖
- 需要手动补充很多信息
- skill 输出不够用

**使用问题特征**：
- 没有用对 skill
- 用了 skill 但没用对方式
- 可以更高效但没做到

**协作问题特征**：
- 描述问题不够清晰
- 没有提供足够上下文
- 沟通轮次过多

**代码问题特征**：
- 项目本身的 bug
- 需求不明确
- 实现偏差

### Step 3: 给出建议

针对每种问题类型，给出具体建议：

```markdown
## 诊断报告

### 问题 1: [问题描述]
- 类型：配置问题/使用问题/协作问题/代码问题
- 分析：[为什么会这样]
- 建议：[具体优化方式]

### 问题 2: ...
```

### Step 4: 记录到文档

将诊断结果保存到：`docs/plans/YYYY-MM-DD-session-diagnosis.md`

```markdown
# 协作复盘诊断报告

## 项目信息
- 项目/功能：xxx
- 时间：xxx
- 使用技能：xxx

## 诊断结果

### 配置问题
- 问题：xxx
- 建议：xxx

### 使用问题
- 问题：xxx
- 建议：xxx

### 协作问题
- 问题：xxx
- 建议：xxx

### 代码问题
- 问题：xxx
- 建议：xxx

## 下一步行动
- [ ] 优化 xxx skill
- [ ] 调整使用方式
- [ ] 改进沟通方式
```

### Step 5: 执行优化

根据诊断结果：
- 如果是**配置问题** → 询问用户是否要现在优化相关 skill
- 如果是**使用问题** → 告知用户更好的使用方式
- 如果是**协作问题** → 告知用户更好的沟通方式
- 如果是**代码问题** → 建议用户后续注意

## 诊断技巧

### 引导用户回顾

- "过程中最困扰的问题是什么？"
- "有没有感觉用某个 skill 不太顺手的？"
- "有没有感觉跟我说的问题总是理解不对？"
- "有没有感觉某个问题修了好几次？"

### 识别问题类型

- **配置问题** → "这个场景 skill 好像没覆盖到"
- **使用问题** → "其实可以用另一个 skill 更方便"
- **协作问题** → "你要是提前这么说我就懂了"
- **代码问题** → "这是项目本身的实现问题"

## 输出要求

1. 诊断要具体，不能泛泛而谈
2. 每条建议要可执行
3. 保存诊断报告到文档
4. 询问用户是否要立即执行优化
